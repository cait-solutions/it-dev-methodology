# /roadmap — Value-ranked приоритизация по North Star

> **Цель:** ранжировать кандидатов проекта по **ROI к North Star** (минимум усилий → максимум вклада в цель) и порекомендовать порядок работы. Лекарство от «делаем ради делаем»: каждая задача получает измеримую оценку вклада, а не интуитивный `High/Medium`. Применима к **любому** проекту-консьюмеру, включая enabling-проекты (инфраструктура/методология).

**Что делает:** читает `NORTH-STAR.md` (критерий ценности) + кандидатов из `ROADMAP.md`, форсирует явную оценку Impact/Confidence/Effort на каждого, считает RICE-score, выводит ранжированный список с обоснованием. **Рекомендует** — решение приоритетов остаётся за владельцем (Граница 12).

**Что НЕ делает (границы):**
- ❌ **Не генерирует кандидатов** — ранжирует уже существующих в `ROADMAP.md`. Генерация новых из сигналов = `/vision review` (IDEAS → Considered). `/roadmap` — следующий этап: Considered → упорядоченный по ценности.
- ❌ **Не тянет live-данные** из CRM / аналитики / внешних сервисов. North Star — декларативный, self-reported (Граница 3/11). Методология ведёт приоритизацию, не исполняет и не хранит runtime-метрики.
- ❌ **Не решает за владельца** — выводит ранжирование + рекомендацию; reorder/аннотация ROADMAP только по явному `y` (Граница 12).

---

## Рекомендуемая модель

**Default tier (Sonnet)** — основной выбор. Оценка Impact/Effort + синтез ранжирования требуют reasoning.

**Upgrade to Capable (Opus) if:**
- 15+ кандидатов одновременно (большой набор для согласованного ранжирования)
- Enabling-проект с нетривиальным leverage-расчётом через несколько growth-проектов

**Downgrade to Fast (Haiku):** ❌ не рекомендуется — это аналитическая команда, не навигационная.

**Mid-task escalation:** нет (single-pass ранжирование).

**Pre-flight model check:** **да** — спроси пользователя какая модель активна (или используй подтверждённую в сессии), сравни с Default. Mismatch ≥ 2 ступени → пауза + рекомендация. См. `.claude/model-tiers.md`.

## Шаг 1 — Прочитать North Star

Прочитать `NORTH-STAR.md` в корне проекта.

**Если `NORTH-STAR.md` не существует** (graceful, не падение) — это первый запуск value-orientation на проекте:

```
⚠️ NORTH-STAR.md не найден — нет критерия для ранжирования по ценности.

Это целевая функция проекта (VISION Ось 8). Чтобы /roadmap мог ранжировать,
нужно зафиксировать North Star. Создать сейчас? Я задам несколько вопросов.
```

При согласии — **опрос ключевых параметров с рекомендацией-дефолтом на каждый** (Recommendation-first rule). По одному вопросу за раз:

1. **Главная метрика ценности?** → Рекомендация-дефолт: «рост — выручка/MRR (рекомендуется для коммерческого проекта)». Альтернативы: число платящих клиентов · лиды/мес · конверсия · нефинансовая (время до результата / охват) для enabling/внутреннего.
2. **Project role — growth или enabling?** → Рекомендация по контексту: проект напрямую продаёт → `growth`; инфраструктура/инструмент/методология → `enabling`. Если `enabling` — спросить какие growth-проекты он обслуживает (dependency-edges).
3. **Target и горизонт?** → если владелец не знает — записать `<не задан>` и пометить, не блокировать (ранжирование возможно и по относительному Impact).
4. **Baseline / текущее значение?** → опционально.

После ответов — создать `NORTH-STAR.md` из `templates/NORTH-STAR.template.md`, заполнив ответы, проставив дату состояния = сегодня. Затем перейти к Шагу 2.

**Если `NORTH-STAR.md` существует** — прочитать. Извлечь: метрику, target, current state + **дату обновления состояния**, `project_role`, dependency-edges (если enabling).

### Staleness-check

Сравнить «Дата обновления состояния» с сегодня. Если старше **30 дней** (порог по умолчанию):

```
⚠️ Состояние North Star обновлялось <N> дней назад (<дата>).
   Ранжирование по устаревшему состоянию вводит в заблуждение.
   Обновить текущее значение метрики перед ранжированием? (да — задам вопрос / нет — ранжирую как есть)
```

При «да» — спросить текущее значение, обновить `NORTH-STAR.md` (значение + дата = сегодня). При «нет» — продолжить, но в финальном выводе пометить `⚠️ ранжирование по состоянию от <дата>`.

## Шаг 2 — Собрать кандидатов из ROADMAP.md

Прочитать `ROADMAP.md`. Кандидаты для ранжирования = записи из секций **`## Considered`** (High/Medium/Low) и **`## Next`**. Секции `## Now` (уже в работе), `## Done`, `## Rejected`, `## On hold` — **не** ранжируются.

- `ROADMAP.md` отсутствует или нет кандидатов → graceful:
  ```
  Нет кандидатов для ранжирования (ROADMAP.md пуст или нет записей в Considered/Next).
  Сначала наполни backlog: /vision review (сигналы IDEAS → Considered).
  ```
  Завершить.

Сформировать список кандидатов (название + краткое описание из ROADMAP).

## Шаг 3 — Форсинг оценки (Impact / Confidence / Effort)

Для **каждого** кандидата оценить три оси. ⛔ **Forcing function** — нельзя написать число без обоснования-предложения; формальная оценка не зачтена.

**Impact (1-10) — вклад в North Star:**
- Для `growth`-проекта: насколько кандидат двигает собственную метрику ценности. 10 = прямой крупный driver метрики, 1 = почти не влияет.
- Для `enabling`-проекта: **leverage** — насколько кандидат ускоряет/разблокирует growth-проекты из dependency-edges. 10 = разблокирует ключевой growth-проект, 1 = внутреннее удобство без внешнего эффекта.
- ⛔ Каждая оценка Impact — с одним предложением: *почему* это число (механизм влияния на метрику).

**Confidence (0-100%) — уверенность в оценке Impact:**
- Насколько уверены что Impact реализуется. Есть данные/прецедент → высоко; гипотеза → низко.
- ⛔ Обоснование: на чём основана уверенность.

**Effort — размер работы:** `S` (1) · `M` (3) · `L` (8) очков (фибоначчи-шкала, не часы — оценка относительная).
- ⛔ Обоснование: что входит в работу (откуда размер).

**RICE-derived score:**
```
Score = (Impact × Confidence%) / Effort
```
(Reach свёрнут в Impact — «вклад в North Star» уже включает охват × эффект.)

**Few-shot — хорошо / плохо (anti-perfunctory):**

✅ **Хорошо:**
> «Лендинг с ценой» — Impact **8**: цена на странице снимает главный барьер конверсии, прямой driver платящих (метрика проекта). Confidence **70%**: A/B на похожем продукте дал +30% (прецедент, но не на нашей аудитории). Effort **M (3)**: одна страница + копирайтинг, без бэкенда. **Score = (8 × 0.70) / 3 = 1.87**.

❌ **Плохо:**
> «Лендинг» — Impact 8, Confidence 90%, Effort S. Score высокий. — *нет обоснования ни по одной оси, числа из воздуха = не зачтено.*

❌ **Плохо (enabling без leverage):**
> Для методологии: «новый валидатор» — Impact 9 потому что полезно. — *«полезно» не leverage; нужно: какой growth-проект и как ускоряет.*

Свести в таблицу:

| Кандидат | Impact (почему) | Conf% (на чём) | Effort (что входит) | Score |
|---|---|---|---|---|
| ... | 8 — ... | 70 — ... | M(3) — ... | 1.87 |

## Шаг 4 — Ранжированный вывод + рекомендация

Отсортировать кандидатов по **Score (убывание)** — это «минимум усилий → максимум роста».

Вывести пользователю:

```
## Value-ranked roadmap — <проект> (North Star: <метрика>, <target>)
<если staleness не обновлён: ⚠️ ранжирование по состоянию от <дата>>

1. <кандидат>  — Score 1.87  (Impact 8 · Conf 70% · Effort M)
   Почему вверху: <одно предложение>
2. ...

### Рекомендация
Топ-1-2 по ROI: <кандидаты>. Минимум усилий, максимум вклада в <метрика>.
<если есть low-score высокого Effort: «Отложить/пересмотреть: <кандидат> — высокий Effort при низком вкладе».>
```

Для `enabling`-проекта добавить строку: *«Ценность считалась через leverage на growth-проекты: <список из dependency-edges>».*

## Шаг 5 — Аннотация ROADMAP (только по подтверждению)

⛔ **Граница 12 — не решать за владельца.** Спросить:

```
Записать score-аннотации в ROADMAP.md Considered (рядом с каждой записью: `[score: N.NN — /roadmap YYYY-MM-DD]`)?
Это поможет видеть приоритет в самом backlog. Порядок записей НЕ меняю — только аннотирую. (y / n)
```

- **`y`** → для каждого ранжированного кандидата добавить в его запись в `ROADMAP.md` строку-аннотацию `*Score:* N.NN — /roadmap YYYY-MM-DD (Impact N · Conf N% · Effort X)`. **Не переупорядочивать** секции, не менять текст записей — только добавить аннотацию (single-source ROADMAP сохранён, владелец сам решает физический порядок).
- **`n`** → ничего не писать, ранжирование остаётся в выводе чата.

> **Почему аннотация, а не reorder:** `ROADMAP.md` — single-source, его структуру (Now/Next/Considered) ведёт владелец + `/vision review` + `/plan`. `/roadmap` добавляет слой ценности (score), но не перехватывает владение порядком. Это соблюдает Границу 12 и Quality bar 9.

## Граница с /vision review (не дублировать)

| Команда | Этап | Вопрос |
|---|---|---|
| `/vision review` | сигналы → реестр | «что вообще рассматривать?» (IDEAS/PRODUCT-GAPS → Considered) |
| `/roadmap` | реестр → порядок по ценности | «что из Considered делать первым по ROI к North Star?» |

`/roadmap` **читает** то, что `/vision review` наполнил. Не генерирует кандидатов сам.
