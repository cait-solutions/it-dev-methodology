# /ux-audit — Структурный аудит UX существующих флоу

> **Цель:** структурированный аудит **UX существующего продукта** — замапить ключевые пользовательские флоу, найти friction-points, сравнить с эталонами лучших практик, выдать приоритизированный список находок. Сосед `/architecture-audit` (архитектура), `/doc-audit` (документация), `/product-check` (PRODUCT.md ↔ код) — не `/web-design` (тот проектирует НОВОЕ, этот оценивает СУЩЕСТВУЮЩЕЕ).

**Когда запускать:**
- Подозрение на UX-friction без конкретного зафиксированного бага (пользователь "как-то неудобно", но не может назвать точку).
- Перед крупным редизайном флоу — baseline текущего состояния.
- Periodic health-check UX (нет автоматического триггера в `/plan` — manual, как `/doc-audit`).
- Manual triggered (запрос пользователя / владельца продукта).

**НЕ дублирует соседей:** `/architecture-audit` смотрит на структуру кода/карт, не на то как флоу *ощущается* пользователем. `/doc-audit` — механическая свежесть документов. `/product-check` — соответствие PRODUCT.md коду, не качество UX. `/web-design` — проектирование нового экрана/флоу с нуля, не оценка того что уже есть.

**ЗАПРЕЩЕНО:** менять код/вёрстку автоматически. Только флоу-карта + находки + приоритизация. Human review required перед любым фиксом.

---

## Рекомендуемая модель

**Extended (UI settings):** effort: **High** · thinking: **ON** — оценка friction и сравнение с эталонами требует reasoning, не механический прогон.

**Default tier:** **Default (Sonnet)**. Основной выбор для флоу-мапинга и приоритизации находок.

**Upgrade to Capable tier (Opus) if:**
- Флоу пересекает 3+ сервиса/компонента (сложная cross-cutting оценка friction)
- Найдено ≥5 friction-points одновременно — синтез приоритизации нетривиален

**❌ Fast tier:** НЕ рекомендуется — риск поверхностной оценки без реального сравнения с эталоном.

**Mid-task escalation:** нет (single-pass анализ).

**Pre-flight model check:** да — при старте команды спроси какая модель активна, сравни с Default tier.

---

## Шаг 0 — Scope

1. Какой флоу (или набор флоу) аудируем? Если не указано пользователем — спросить: "Какой пользовательский путь смотрим? (например: onboarding, поиск+фильтр+детали, checkout)".
2. Определить **акторов** флоу (кто проходит этот путь — из USER-MAP.md если существует, иначе спросить).
3. Определить **эталоны сравнения** — best-in-class продукты того же класса задачи (напр. Linear/Attio для B2B list-detail UI). Если пользователь не назвал — предложить 1-2 общепринятых для этого класса UI, спросить подтверждение.

---

## Шаг 1 — Flow-map

Пройти флоу шаг за шагом (по коду/UI, не по памяти — Read-Grounding принцип из `/plan` применим и здесь):

| Шаг флоу | Действие пользователя | Ожидаемый результат | Фактическое поведение (если отличается) |
|---|---|---|---|
| 1 | ... | ... | ... |

Список шагов ДОЛЖЕН включать edge-точки, не только happy path: пустой результат, ошибка, повторный визит, отмена на середине.

---

## Шаг 2 — Friction-points

Для каждого шага флоу — применить heuristics (HICCUPPS-подобные, как в связанном testing-дисциплине паттерне):
- **Consistency** — ведёт ли себя как остальные похожие элементы в этом же продукте?
- **Comparison** — как этот же шаг решён в эталоне из Шага 0?
- **User expectations** — что пользователь ожидал бы по общему опыту с похожими продуктами?
- **Purpose** — решает ли шаг задачу пользователя за минимум усилий?

**Mandatory output** (S-028-подобно — нельзя написать "всё ок" без таблицы):

| Шаг флоу | Friction? | Описание | Сравнение с эталоном |
|---|---|---|---|
| ... | да/нет | ... | ... |

Пустой список friction-points = явно написать "0 найдено" с обоснованием (не пропуск шага).

---

## Шаг 3 — Приоритизация находок

Для каждого найденного friction-point:

| Находка | Severity (🔴/🟡/🟢) | Затронутые пользователи (все / edge case / первый визит) | Оценка усилия фикса | Рекомендация |
|---|---|---|---|---|

Severity: 🔴 — блокирует задачу пользователя; 🟡 — раздражает, но обходится; 🟢 — косметика.

---

## Шаг 4 — Вывод

Показать пользователю:
1. Flow-map (Шаг 1) целиком.
2. Friction-points таблицу (Шаг 2) целиком.
3. Приоритизированный список (Шаг 3), отсортированный по Severity.
4. Явная рекомендация: топ-3 находки для следующего `/plan`, если пользователь хочет фиксить.

Находки severity 🔴 → предложить сразу завести `/plan` для фикса (не просто список, действие).

**→ `/test`** — если во время аудита всплыл функциональный **баг** (не friction, а «сломано»: 500, краш, неверная логика, вёрстка поехала) → это ось `/test`, веди находку в CODE-GAPS через него, не в UX-приоритизацию.
