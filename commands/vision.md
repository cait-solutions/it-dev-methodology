# /vision — Продуктовая оптика (strategy · review · sync)

> **Цель:** единая точка входа для работы с продуктовым видением. Три режима — стратегия (куда продукт идёт), ревью сигналов (что строить дальше), сверка (vision ↔ реальность). Заменяет `/product-vision`, `/product-review`, `/sync-vision` (consolidation v6.0.0). НЕ для аудита точности PRODUCT.md — это `/product-check` (механический, отдельная команда).

## Диспетчер режимов

Прочитать аргумент `$ARGUMENTS` (первое слово) и выбрать режим. Если режим не указан — спросить пользователя какой из трёх, дав рекомендацию по триггеру (см. ниже).

| Режим | Алиасы | Отвечает на вопрос | Был | triggers-ключ |
|---|---|---|---|---|
| **strategy** | `strat`, `axes` | Куда продукт должен двигаться через 6-12 мес? Стратегические оси на 1-3 года. | `/product-vision` | `last_product_vision` |
| **review** | `signals`, `ideas` | Что болит/что строить дальше? IDEAS + PRODUCT-GAPS → приоритизация → ROADMAP. | `/product-review` | `last_product_review` |
| **sync** | `reconcile` | Vision устарел? Код дрейфует? Двусторонняя сверка VISION/ADR ↔ код. | `/sync-vision` | `last_sync_vision` |

**Recommendation-first при отсутствии аргумента** (closes G-102): сначала дать рекомендацию по триггеру, потом спросить.
- 10+ inbox / 5+ открытых OQ / Type-C подозрение / 90+ дней без апдейта → рекомендовать **sync**.
- Накопились IDEAS/PRODUCT-GAPS, прошло ~14 дней, триггер из /plan → рекомендовать **review**.
- Конец квартала / стратегический разговор / 1-2 квартала с последней стратегии → рекомендовать **strategy**.

**Граница режимов (не дублируют scope):**
- **strategy** vs **review**: strategy = «куда двигаться через год» (оси, новые категории); review = «что болит сейчас» (тактические сигналы). Кандидат не прошёл фильтр оси → review/ROADMAP, не strategy/VISION.
- **sync** ≠ strategy/review: sync — двусторонняя сверка факта (vision устарел ИЛИ код дрейфует), не генерация нового. Конфликт → OQ, не решение.

## Рекомендуемая модель (per-mode)

**Extended (UI settings):** все режимы /vision — reasoning/синтез → effort: **High** · thinking: **ON**. См. `.claude/model-tiers.md` § Effort & Thinking.

Каждый режим — свой tier. Pre-flight model check **обязателен во всех трёх** — спроси пользователя какая модель активна (или используй ранее подтверждённую в сессии); при mismatch — пауза + рекомендация (порог и формат: `.claude/model-tiers.md § Pre-flight model check`; under-powered — любая ступень вниз).

| Режим | Default tier | Upgrade | Downgrade | Mid-task escalation |
|---|---|---|---|---|
| **strategy** | **Capable** | (всегда Capable — верх) | никогда (слабая модель → axes теряют strategic depth) | нет (single deep pass) |
| **review** | Default | 20+ unreviewed IDEAS за период (паттерны через большой набор) | Fast если < 5 IDEAS (короткий обзор) | нет (single pass) |
| **sync** | Default | 10+ inbox И 5+ открытых OQ; обнаружен Type C конфликт | никогда (конфликтный анализ требует reasoning) | нет (single-pass A-E + report) |

> **Pre-flight:** strategy запущен не на Capable → 🔴 strongly recommend upgrade перед началом. review/sync на Capable — допустимо (overhead, не блок).

## Mode: strategy — Стратегическая оптика продукта

> Определить **стратегические оси развития продукта** на 1-3 года вперёд. Декомпозиция категорий ценности, anti-anchoring расширение, ranking по leverage. Обновляет VISION.md → Активные оси. Запускать вручную раз в 1-2 квартала.

### Шаг -1 — Калибровка уровня оси

Прежде чем называть что-то "осью" — фильтр:

1. **Это меняет КАТЕГОРИЮ продукта или улучшает текущую?** — Только первое = ось
2. **Через сколько перестанет быть актуальной?** — < 6 мес → фича; 6-18 мес → среднесрочное; 1-3 года → настоящая ось
3. **Это inherent quality (надёжность, скорость)?** — Не ось → quality bar в CLAUDE.md или ROADMAP
4. **Если убрать ось — продукт остаётся тем же?** — Тем же = улучшение, не ось
5. **Ось открывает новый сегмент пользователей или контекст?** — Нет → крупная фича, не ось

Кандидаты не прошедшие фильтр → ROADMAP, не VISION.

### Шаг -0.5 — Принудительное расширение (anti-anchoring)

Прежде чем смотреть архитектуру — ответь свободно:
- Какие категории продуктов смежны с текущим?
- В какие категории мог эволюционировать через 1-2 года?
- Какие сегменты сейчас не обслуживаются?
- Какие контексты использования могли появиться?

**Архитектурные ограничения = СТОИМОСТЬ оси, не критерий отсева.**

### Подготовка

Прочитать: 1. PRODUCT.md (текущий контракт) · 2. DEVLOG.md (последние 2-3 месяца) · 3. IDEAS.md (все записи) · 4. VISION.md (текущие активные оси) · 5. ROADMAP.md (что уже в работе).

### 1. Структурные оси роста

Какие 3-5 направлений могут расширить продукт? Для каждой прошедшей фильтр: что это (1 предложение) · какую новую категорию задач откроет · какие ранние сигналы в IDEAS/DEVLOG · что нужно изменить в архитектуре (стоимость).

### 2. Декомпозиция

Для каждой оси — 3 уровня: **Уровень 0** название · **Уровень 1** 2-4 крупных блока · **Уровень 2** 2-3 конкретных функции на блок.

### 3. Гипотезы о ценности

Для каждой оси: кому нужно (конкретный сценарий) · какая боль закрывается · новый сегмент или улучшение · что произойдёт если НЕ делать в течение года.

### 4. Ranking

Основные: **стратегическая важность** (что определяет будущее, что нельзя добавить потом) · **рычаг** (какая ось разблокирует больше других). Поддерживающие: сигналы из реальности · готовность к реализации. **Top-2 оси** → кандидаты на детальное планирование в ROADMAP.

### 5. Anti-roadmap

**Стратегические границы (долгоживущие):** «Продукт остаётся X пока не решён Y». Снимается только явным решением.
**Тактические откладывания (краткосрочные):** «Не делаем Z пока не проработаны риски W».

### Формат ответа

5-7 страниц связного текста, не таблицы. Стратегический документ, не to-do list.

### После анализа (strategy)

Спросить: "Перенести предложенные оси в VISION.md → Активные? (y/n/selective)". Ждать ответа. Не переносить самостоятельно.

**ROADMAP Mermaid sync:** если подтверждены новые оси → VISION.md + ROADMAP.md получают новые Considered-записи → обновить mermaid-блок в секции `## Визуальный roadmap` + `bash scripts/update-mermaid-links.sh <roadmap-path>`. Если mermaid-блока нет — добавить по шаблону `templates/ROADMAP.template.md`.

Обновить triggers.json: `last_product_vision = { date: today, plans_since: 0 }`

## Mode: review — Продуктовый анализ сигналов

> Обработать накопленные IDEAS.md (raw signals) + PRODUCT-GAPS.md (classified gaps) → приоритизировать → перенести actionable в ROADMAP. Reviewed IDEA = product gap → конвертация в P-NNN запись. Запускается по триггеру из /plan или вручную раз в 14 дней.

### Подготовка

Прочитать: 1. **IDEAS.md** (raw signals) · 2. **PRODUCT-GAPS.md** если существует (P-NNN записи: feature/capability/ux/integration/edge-case с severity) · 3. DEVLOG.md (последние 14 дней) · 4. VISION.md (активные оси).

Два артефакта-сигнала: **IDEAS** = raw (observation без structure) · **PRODUCT-GAPS** = classified (structured gap с severity/категорией/гипотезой/potential fix).

Workflow: reviewed IDEAS → confirmed actionable gap → конвертируется в новую P-NNN запись в PRODUCT-GAPS.md → сортируется по severity → попадает в ROADMAP.

Если оба пусты → "IDEAS.md и PRODUCT-GAPS.md не заполнены. Анализ невозможен без данных."

### Четыре вопроса

**1. Friction patterns:** что пользователь делает руками что могло быть автоматическим? (сигнал: одна команда в IDEAS ≥ 3 раза за период)
**2. Discovery gaps:** какие функции есть но не используются? (сигнал: фича в DEVLOG, не используется неделями после деплоя)
**3. Visibility gaps:** что происходит внутри что пользователь не видит? (сигнал: записи "я не знал что..." в IDEAS)
**4. Natural extensions:** какие 2-3 фичи естественно вырастают из паттернов? (привязка к конкретным сигналам, не абстракции)

### Формат вывода

5-7 конкретных предложений. Каждое: **Приоритет** (High/Medium/Low) · **Тип** (friction/discovery/visibility/extension) · **Предложение** (конкретное действие) · **Обоснование** (привязка к данным) · **Риски** (обязательно для High/Medium: 2-3 конкретных) · **Ось** (если применимо).

❌ Запрещено: предложения без привязки к данным · High/Medium без блока "Риски".

### После анализа (review)

Спросить: "Перенести High-приоритетные в ROADMAP.md → Considered? (y/n)". Не переносить самостоятельно.

**IDEAS → PRODUCT-GAPS classification:** если reviewed IDEAS описывают product coverage gap (а не workflow friction) — спросить:
```
📋 Reviewed IDEA [описание] описывает product coverage gap.
   Конвертировать в PRODUCT-GAPS запись P-NNN? (y/n)
   - Категория: [feature-gap | capability-gap | ux-gap | integration-gap | edge-case-gap]
   - Severity: [🔴 High | 🟡 Medium | 🟢 Low]
   - Use case (затронут): [конкретный сценарий]
```
При `y` — создать P-NNN в PRODUCT-GAPS.md, в IDEAS пометить `[reviewed:converted-to-P-NNN]`.

**ROADMAP Mermaid sync (PR-coupling):** если перенесены записи в ROADMAP.md (добавлены/изменены/закрыты узлы) — обновить `## Визуальный roadmap`: отредактировать mermaid-блок, затем `bash scripts/update-mermaid-links.sh <roadmap-path>`. Только textual редакция без изменения нодов — не обновлять.

Обновить triggers.json: `last_product_review = { date: today, plans_since: 0 }`

## Mode: sync — Двусторонняя сверка vision ↔ реальность

> Синхронизировать VISION/ADR с реальностью кода (inbox + OPEN-QUESTIONS + Type C конфликты). Two-way check: vision устарел? код дрейфует? Обновить VISION.md / RISKS.md / OPEN-QUESTIONS.md. Запускается по триггеру (10+ inbox, 5+ OQ, Type C, 90+ дней), не регулярно.

**ЗАПРЕЩЕНО:** менять файлы автоматически. Только отчёт и рекомендации (кроме Шага 5 автоматических действий ниже).

### Шаг 1 — Inventory источников

Прочитать VISION.md · все ADR · SKILL.md модулей (если есть) · SYSTEM-MAP.md (если есть) · inbox/ необработанные файлы · grep по ключевым понятиям в коде.

### Шаг 2 — Source validation

Для каждого источника: когда последний раз обновлялся (git blame) · подтверждается реальным кодом? · источники старше 90 дней без активности — отдельно.

### Шаг 3 — Cross-reference анализ (5 типов)

**A** — Реальность кода богаче чем описана в vision · **B** — Vision описывает то чего нет в коде · **C** — Vision и код противоречат · **D** — Источник невалиден (устарел/гипотетический/галлюцинация) · **E** — Vision слишком расплывчат.

### Шаг 4 — Создание отчёта

`docs/sync-vision-reports/YYYY-MM-DD.md`:
```markdown
# Sync Vision Report YYYY-MM-DD
## Контекст
- Триггер: [какой]  · Источников проверено: N
## Type A — Новое знание для vision (N)
## Type B — Tech debt — Блокирующие (N) / Неблокирующие (N)
## Type C — Конфликты требующие решения PM → ссылки на OQ-NNN
## Type D — Отвергнутые источники → причины
## Type E — Vision требует уточнения → OQ-NNN
## Рекомендации — какие ADR обновить/создать (supersedes) · inbox → _processed/rejected · что блокирует
```

### Шаг 5 — Действия

Автоматически (без вопроса): Type B блокирующие → RISKS.md · Type C → OQ записи · Type D → `inbox/_processed/rejected/` · Type E → OQ с пометкой "vision-clarification-needed".
Требует подтверждения: Type A → показать предложенные изменения VISION/ADR, спросить "Применить обновления? (y/n/selective)". ⛔ НЕ меняй VISION/ADR до ответа.

### После завершения (sync)

1. DEVLOG: `[sync-vision] Report YYYY-MM-DD: N type-A, N type-B, N type-C, N type-D, N type-E`
2. triggers.json: `last_sync_vision = { date: today, plans_since: 0 }`
