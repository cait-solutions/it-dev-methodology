# /plan — Универсальная команда планирования

Ты — архитектурный критик проекта. Задача не просто составить план, а найти всё что может пойти не так — до того как написана первая строка кода.

**ЗАПРЕЩЕНО:** писать код, изменять файлы, создавать миграции. Только план, риски, вопросы и checklist.

---

## Рекомендуемая модель

**Strategy:** Default (Sonnet) — основной выбор. Upgrade to Capable (Opus) только при триггерах.

**Default tier (Sonnet):** Основной выбор для всех планов. Достаточна для архитектурного анализа, диагностики, синтеза решений.

**Upgrade to Capable tier (Opus) if:**
- `[contract]` + threat model сложнее стандартного
- `[security]` новый endpoint с нетривиальным threat-моделем
- Multi-service refactor (50+ файлов, много зависимостей)
- Обнаружена системная проблема в диагностике (class-bug в 3+ файлах)
- User explicitly asks for deeper analysis
- **Reasoning-depth detector (closes G-082 — ось глубины, не объёма):** задача по характеру **deep-reasoning**, не scope-bound? Признаки: visual/CSS-баг где «работает по коду но не в DOM», race condition, нетривиальный алгоритм, баг который уже **не сошёлся** за N итераций в прошлой сессии, требование «почему X не работает» без явной причины. Такие задачи **маленькие по объёму но глубокие по reasoning** — scope-триггеры (50+ файлов) их НЕ ловят. Если задача deep-reasoning И текущая модель < Capable → рекомендовать Opus **до начала анализа** (а не ждать пока /code зациклится). Эмпирика G-082: Sonnet циклил sidebar поверхностно, Opus решил за раз через pattern-comparison.

**❌ Fast tier (Haiku):** НЕ рекомендуется для регулярных планов
- Риск: неправильный синтез, упущение консистентности
- Допускается только для grep/validation если у тебя явно нет reasoning-based шагов
- Лучше переплатить на Default чем пропустить архитектурное нарушение

**Mid-task escalation:** нет (анализ в /plan не требует mid-task переоценки как /code)

**Pre-flight model check:** **да — при старте команды** спроси пользователя какая модель активна (или используй ранее подтверждённую в сессии) и сравни с Default tier. Если mismatch ≥ 2 ступени — пауза + рекомендация перед началом анализа. См. `.claude/model-tiers.md` секция Pre-flight model check.

## Шаг -4: Correction check (первый, перед всем остальным)

Прочитай текст нового задания и **оцени семантически**: описывает ли оно ситуацию, где прошлая работа агента была неполной, неверной или что-то важное было пропущено?

**Это НЕ поиск ключевых слов.** Оценивай смысл и контекст:
- Задание содержит неявную или явную критику предыдущего решения?
- Указывает что что-то сделано "не так", "не полностью" или "не системно"?
- Ссылается на несделанное рядом с тем что уже сделано?
- Выражает неудовлетворённость результатом предыдущей итерации?

**Коррекция** (оценивай смысл, не слова): «не нравится что только по этим словам» (критика реализации) · «системное решение не реализовано» (недовольство) · «ты добавил X, а Y — нет» (пропуск) · «хочу пересмотреть как реализовали» (пересмотр).
**НЕ-коррекция:** «добавь X» (новая задача) · «как подойти к Y?» (вопрос) · «расширь Z» (расширение без критики).

Если ДА → **ПЕРЕД** pre-flight классифицировать семантически: **agent gap** или **product gap**?

- **Agent gap (AGENT-GAPS.md):** причина в том что **Я** как агент не подумал / пропустил правило / проигнорировал контекст / принял ложное assumption. Закрывается **правилом в методологии**.
- **Product gap (PRODUCT-GAPS.md):** причина в том что **продукт** не покрывает функцию / use case / edge case. Закрывается **/plan для фичи**.
- При сомнении → AGENT-GAPS (можно reclassify в /retro).

Затем — **автоматически записать** без вопроса к пользователю:

**Шаг A — Дедуп-grep** (перед записью):
```bash
grep -i "<2-3 ключевых слова из сути коррекции>" AGENT-GAPS.md   # для agent
grep -i "<2-3 ключевых слова>" PRODUCT-GAPS.md                    # для product
```
- **Hit** → "📝 Похожий gap `G-NNN` уже зафиксирован — дублировать не буду." → перейти к Шагу -3.
- **Нет hit** → перейти к Шагу B.

**Шаг B — Запись**:
Определить следующий ID (G-NNN / P-NNN), добавить запись **сверху** в секцию `## Записи` целевого файла. Формат — из AGENT-GAPS.md.template / PRODUCT-GAPS.md.template.

Поле `Контекст:` = `/plan`.

**Шаг C — Извещение + opt-out** (одна строка после записи):
```
📝 Записано: [G-NNN | P-NNN] — [краткое что пропустил / что не покрывает]. Отменить: напиши "нет" / "n".
```
Если пользователь отвечает "нет" / "n" — удалить только что добавленную запись (И откатить инкремент Шага D). Затем перейти к Шагу -3.

**Шаг D — Session gap counter + real-time эскалация (слой-3, closes класс «серия однотипных gap'ов невидима в моменте»):**

После записи gap'а (Шаг B) — инкрементировать `session_gap_counter` в `.claude/state/triggers.json` и проверить порог. Это ловит серию однотипных провалов **в реальном времени** (твоя CSS-сессия: 3 assumption-gap подряд), в отличие от `recurrence_rate` в `/architecture-audit` (пост-фактум через ≥5 планов).

1. Прочитать `triggers.json` → `session_gap_counter` (defensive: `.get('session_gap_counter') or {'session_marker': None, 'counts': {}}` — отсутствие поля НЕ ошибка, старый consumer до schema-бампа).
2. **Session-boundary (timestamp-прокси — в Claude Code нет явного session-id команде):** сравнить текущее время с `session_marker`. Если `session_marker` отсутствует ИЛИ разрыв > `gap_session_window_hours` (default **6ч** из `CLAUDE.local.md ## Iteration watchdog`) → **новая сессия**: сбросить `counts = {}`. Иначе — та же сессия, продолжаем накопление.
3. Обновить `session_marker` = текущее время (ISO8601). Инкрементировать `counts[<категория gap'а>] += 1` (категория = `assumption-gap`/`logic-gap`/`completeness-gap`/etc — то же поле что в записи gap'а).
4. Сохранить `triggers.json`.
5. **Порог-эскалация (one-shot per категория за сессию):** если `counts[<категория>]` достигла `gap_escalation_threshold` (default **3**) И эскалация для этой категории ещё не показана в этой сессии (трек через `counts["_escalated_<категория>"] = true`) → вывести **пользователю**:
   ```
   🛑 SESSION GAP PATTERN: 3-й <категория> gap за эту сессию (<G-NNN>, <G-MMM>, <G-KKK>).
      Это сигнал что подход к этому классу задач не сходится — не разовая ошибка, а паттерн.
      РЕКОМЕНДАЦИЯ:
      • Останови накопление однотипных провалов — смени подход к этому классу (не «ещё одна попытка»).
      • Для deep-reasoning класса (CSS/DOM, race, алгоритм): начни новую сессию на Capable (Opus) +
        примени pattern-comparison с рабочим эталоном (G-082), не итерируй на текущей модели.
      • Порог настраивается: CLAUDE.local.md ## Iteration watchdog → gap_escalation_threshold.
   ```
   Затем `counts["_escalated_<категория>"] = true`, сохранить. Не спамить на 4-м, 5-м gap'е той же категории за сессию.

Defensive: если `triggers.json` отсутствует/невалиден → пропустить Шаг D тихо (gap уже записан в Шаге B, counter — дополнительный сигнал, не блок).

Если пользователь отвечает "нет" / "n" — удалить только что добавленную запись (Шаг C). Затем перейти к Шагу -3.

**Файл не существует** → пропустить тихо (gracefully), перейти к Шагу -3.

Если НЕТ (коррекция не обнаружена) → продолжить к Шагу -3.

Каждый триггер — **предложение с подтверждением**, никогда не авто-запуск.

### Подшаг -0.4 — Hook liveness check (closes mechanism #3 «watchdog не запускался» + G-087 рекурсивная дыра)

Прочитать `.claude/settings.json`. Проверка — **физическое наличие на диске** каждого hook-файла на который ссылается settings.json (не просто «wired в конфиге»). Это closes G-087: settings может корректно ссылаться на хуки, но сами файлы (включая `run-hook.sh`) отсутствуют → все хуки молча падают.

**Извлечь имена hook-файлов — ДВА паттерна** (зеркало canon `auto-update-watchdog.template.py:211-215` — менять синхронно, иначе drift между детекторами):
1. прямой вызов: `.claude/hooks/X` (ловит `run-hook.sh`, `hook-liveness.sh`, любой `.py`/`.sh`)
2. через wrapper: `run-hook.sh X` (ловит `X.py` запускаемые раннером)

Для каждого извлечённого имени — проверить `test -f .claude/hooks/<имя>`. **Обязательно включить `run-hook.sh`** в проверяемый набор (он попадает через паттерн 1 из строки `.claude/hooks/run-hook.sh`).

- **Любой referenced hook-файл (включая `run-hook.sh`) отсутствует на диске** → 🔵 предупреждение (graceful, не hard-block):
  ```
  ⚠️ HOOK DRIFT — settings.json ссылается на отсутствующие hook-файлы:
     • .claude/hooks/run-hook.sh — НЕ найден  (← раннер ВСЕХ хуков: если его нет, все хуки мертвы)
     • .claude/hooks/iteration-watchdog.py — НЕ найден
     Следствие: эти хуки молча падают (sh: No such file) → защита/детекторы/escalation мертвы.
     Причина: методология обновилась, но full sync не прогонялся в этом проекте.
     Рекомендация: запусти `bash <methodology>/scripts/sync-methodology.sh .` затем перезапусти сессию.
  ```
- **Дополнительно — нет `SessionStart` блока ИЛИ нет `auto-update-watchdog` в нём** → та же graceful-рекомендация про sync (auto-update не запустится).
- **Все referenced хуки на диске И SessionStart присутствует** → тихо продолжить.

**Почему здесь, а не только в `check_hook_health` (рантайм-хук) — рекурсия:** `check_hook_health` живёт ВНУТРИ `auto-update-watchdog.py`, который запускается через `run-hook.sh`. Если отсутствует САМ `run-hook.sh` — watchdog не стартует → `check_hook_health` не вызывается → дыру некому детектить (детектор отсутствующих хуков сам недоступен). Slash-команда `/plan` читается агентом гарантированно (Lite И Full), независимо от hook-подсистемы → надёжный floor когда подсистема мертва целиком. Комплементарно три fate-independent детектора: **`hook-liveness.sh`** (SessionStart, pure-sh, БЕЗ run-hook.sh — ловит missing-files когда run-hook.sh мёртв) → **`check_hook_health`** (runtime — ловит когда хуки живы) → **этот /plan check** (always-read — ловит когда подсистема не запущена вовсе). Плюс delivery-слой: parity-gate (/review) → merge_settings_json (sync).

Если `.claude/settings.json` отсутствует → пропустить тихо (bootstrap-ситуация, её ловит сам watchdog при появлении).

## Шаг -1 — Диагностика

**1. Симптом или причина?**

**2. Код или промпт?**

**3. N-й фикс одной проблемы? (domain-aware — closes S-1 / push-кластер урок)**
- grep DEVLOG **двумя проходами** за 7-30 дней:
  ```bash
  grep -oE "\[fix:[a-z-]+\]" <DEVLOG> | sort | uniq -c        # точный тег (fallback для старых записей)
  grep -oE "\[domain:[a-z-]+\]" <DEVLOG> | sort | uniq -c     # семантический домен (D6 domain-indicator)
  ```
- **Точный `[fix:X]` ≥ 2** ИЛИ **`[domain:X]` ≥ 2** (любой из двух) → ⛔ СТОП → запустить `/diagnose`.
- **Почему domain-проход:** кластер симптомов одного корня имеет разные surface-теги (`consumer-push`/`deploy-push`/`command`), но один `[domain:git-push]` → точный grep их не сгруппирует, domain grep — да. Урок push-кластера (v5.19-5.24): 3 фикса домена прошли как разные `[fix:X]` → /diagnose не сработал → корень P-006 назван на 9-м симптоме. См. CLAUDE.md «Domain-indicator для [fix:X]».
- Если `[domain:X]` ≥ 2 — показать domain-related фиксы **явной таблицей** (видимый detection, не молча): `\| дата \| [fix:X] \| [domain:Y] \|`.
- Старые DEVLOG без `[domain:X]` → только точный-тег проход (graceful fallback, не хуже прежнего).

**4. Точечное или структурное решение?** *(явно ответить — не пропускать)*

### Forward-Failure Analysis (обязательный — industry best practice: FMEA + JTBD)

> **Принцип (Gawande):** «подумай о будущем» — aspirational, заполняется формально. Эти три блока — discipline-creating: каждый требует **конкретного ответа который нельзя написать не подумав**. Заменяет интуицию структурой проверенных классов проблем.

**A. FMEA — Failure Mode Enumeration (заменяет «подумай о рисках»):**

> **Глоссарий (читать перед заполнением таблицы):**
> - **FMEA** (Failure Mode and Effects Analysis) — «анализ режимов и последствий отказов». Промышленный метод (NASA, авто): перечисли как решение сломается **до** запуска, пока ещё можно исправить.
> - **S** (Severity / Тяжесть) — насколько плохо если сломается. 10 = потеря данных / полный обход решения, 1 = косметика.
> - **O** (Occurrence / Частота) — насколько часто будет происходить. 10 = почти всегда, 1 = редкий edge-case.
> - **D** (Detection / Обнаружимость) — ⚠️ **обратная шкала**: 10 = **никто не заметит** пока пользователь сам не пожалуется (тихий провал, самое опасное), 1 = тест/валидатор ловит сразу автоматически.
> - **RPN** (Risk Priority Number) = S × O × D. Чем выше — тем приоритетнее fix. Порог: RPN > 200 → обязательная mitigation.

Назови **2-3 конкретных способа** как это решение **не сработает** в реальном использовании (не гипотетически — конкретный механизм поломки). Для каждого — три оценки 1-10:

| Failure mode | S (Тяжесть) | O (Частота) | D (Необнаружимость) | RPN = S×O×D |
|---|---|---|---|---|
| [конкретный способ поломки] | [1-10] | [1-10] | [1-10] | [произведение] |

- **S (Severity / Тяжесть):** насколько критично если сломается (10 = пользователь теряет данные / полный обход решения; 1 = косметика).
- **O (Occurrence / Частота):** насколько вероятно при текущем коде/использовании (10 = почти всегда; 1 = редкий edge).
- **D (Detection / Необнаружимость):** **поймаем ли автоматически ДО пользователя** (10 = никто не заметит пока пользователь не пожалуется; 1 = тест/валидатор ловит сразу). ⚠️ Высокий D — самый опасный: тихий провал.

**Правила:**
- ⛔ **RPN > 200** (любой failure mode) → обязательная mitigation в Шаге 3 «Риски» ИЛИ explicit «accepted risk» с обоснованием. Не игнорировать.
- ⛔ **D ≥ 7** (плохо детектируется) → обязателен шаг который добавляет detection (валидатор / grep-проверка / тест), ИЛИ явное «detection невозможна потому что …».
- Канонический антипример: «валидация наличия файла без проверки содержимого» — S=7 (пользователь уверен «OK»), O=8 (частый кейс), D=9 (валидатор говорит OK, никто не замечает что артефакт stale). RPN=504 → отдельный шаг обязателен. *(closes G-073)*

**B. JTBD Struggling Moment (заменяет «подумай об UX»):**

Ответь **одним конкретным предложением** (нельзя «всё удобно»):
> «Пользователь применил это решение. В какой **конкретный момент** он скажет "проще руками" и обойдёт механизм? Что именно создаёт этот friction?»

- Если ответ «не обойдёт» → обоснуй почему (workflow не оставляет friction-точки).
- Канонический антипример: предложили «запускать скрипт **отдельной командой**» когда логичнее встроить в существующую общую команду — пользователь забудет отдельный шаг, friction = лишнее действие вне привычного workflow. *(closes G-074)*

**C. Integration / non-duplication check (заменяет «не дублируем ли»):**

Если решение добавляет новый механизм (скрипт / шаг / команда / правило):
- [ ] `grep` по существующим командам/скриптам: есть ли уже механизм закрывающий часть этой задачи? Если да — **встроить в него**, не плодить параллельный.
- [ ] Новый артефакт **запускается пользователем отдельно** или **встроен в существующий trigger** (команду/hook/workflow)? Отдельный запуск = требует обоснования «почему нельзя встроить».

⛔ Без заполнения A (таблица RPN) + B (struggling moment) + C (integration check) → план неполный для задач меняющих поведение. Doc-only / typo → `N/A — нет нового механизма/поведения`.

## Шаг 3 — План реализации

⛔ **Antipattern G-050 — НЕ оборачивай шаблон ниже в ` ``` ` code fence при выводе пользователю.**

Шаблон — это **plain markdown**, который рендерится в чате как:
- `##` headers → H2
- `**bold**` → bold
- `---` → горизонтальная линия
- `- [ ]` → checkbox list
- таблицы → таблицы

**Если обернёшь шаблон в code fence** (` ``` `):
- ❌ Bold внутри code не работает (`**Было:**` остаётся как `**Было:**`)
- ❌ Headers интерпретируются как plain text (`## До / После` показывается literally)
- ❌ Визуальная целостность плана разрывается — пользователь видит фрагменты code-block вместо читаемой структуры
- ❌ Class G-046/G-049/G-050: agent ignores own recent rule, действует по памяти про формат «template = code block»

**Шаблон визуально отделён `---` separator и заголовком "Структура финального плана"** — code fence не нужен для маркировки «это template».

✅ **Правильно:**
```
[обычный текст вступления]
---
## Задача
[содержимое]

...
```

❌ **Неправильно:**
```
[обычный текст]
​```​
## Задача
[содержимое]
​```​
```

## Шаг 97 — Sustainment Declaration (Full mode, перед Pre-Mortem)

**Цель:** каждый план, создающий или изменяющий механизм/артефакт, обязан спроектировать его операционный цикл — *кто/что запускает, как актуализируется, как обнаружим что умер/устарел, кто владелец* — и вывести это пользователю **отдельной секцией**. Закрывает класс G-099 (≥10 инстансов: G-047, G-081, G-084, G-087, G-088, G-091, G-092, G-098).

**Триггер: когда выполнять этот шаг**

Full mode — выполнять всегда. Для каждого плана ответь: «создаёт ли или меняет ли этот план lifecycle/семантику хотя бы одного механизма/артефакта?»

Если **ДА** — заполнить таблицу (см. ниже). Если **НЕТ** (doc-only / typo / переформулировка без нового механизма) — одна строка `N/A — причина`.

**Что считается «механизмом / артефактом» (derivation из «Затрагиваемые файлы»):**

Пробеги по каждому файлу из списка «Затрагиваемые файлы» Шага 3 и задай вопрос: «является ли этот файл — или то что он добавляет/меняет — механизмом, который нужно кому-то или чему-то запускать, обновлять, или мониторить?»

Классы артефактов, требующих Sustainment анализа:
- Hook / pre-commit / post-commit
- Script, runner, CLI-команда
- Slash-команда методологии (`/plan`, `/review`, `/sync-audit`, ...)
- Карта (SYSTEM-MAP, USER-MAP, ARTIFACT-MAP)
- Mermaid-ссылка
- Config / triggers.json поле / schema поле
- Registry / manifest (secrets-manifest.yaml, `.code-workspace`, ...)

**Таблица Sustainment Declaration (заполни per-артефакт):**

| Создаваемый / изменяемый артефакт | Trigger (кто/что запускает) | Refresh (как актуализируется при изменениях) | Detection (как обнаружим что умер/устарел) | Owner |
|---|---|---|---|---|
| [имя файла / механизма] | [hook event / команда / cron / ручной] | [sync-methodology / re-run / auto / ручной cadence] | [validate-*.sh --root X / grep / liveness-check / ничего — gap!] | [методология / consumer-проект / пользователь] |

**Правила заполнения:**

- Строки таблицы **деривируются из «Затрагиваемые файлы»** — не свободным ответом. Для каждого файла из списка: класс-check → строка или явный «не механизм».
- **«❌ НЕТ» в любой ячейке** → либо добавь шаг/commitment в план который закрывает этот gap, либо явно: `accepted-gap: [причина почему OK без Detection/Refresh]`.
- **Механизм требующий ручного запуска без trigger-point = gap** → встроить в команду/hook как commitment (closes command-first invariant).

- **Owner glossary:** `методология` = sync-methodology.sh доставляет изменения; `consumer-проект` = consumer применяет сам; `пользователь` = ручной cadence (например, ежеквартальный аудит).
- **Anti-perfunctory:** Owner=методология везде и Detection=N/A везде → красный флаг формального заполнения; если так — напиши явно «self-applying: Detection не нужен потому что [причина]».

**Lite mode:** одна строка `N/A — Lite mode: задача не создаёт механизм (багфикс ≤ 1 файла)`.

## Шаг 98 — Pre-Mortem (Full mode, перед Self-Lint)

⚠️ **Принудительный триггер:** если пользователь выразил сомнение ("точно проверил?") — pre-mortem обязателен в любом режиме.

**Грамматика (Gary Klein):** не «что может пойти не так?» (invites hedging) — а «прошло 3 месяца, фича **уже провалилась**. Почему?» (prospective hindsight, +30% к точности предсказания). Пиши в прошедшем времени как свершившийся факт, не гипотезу.

Напиши **7 сценариев** — каждый с **конкретным механизмом**, не категорией:

1. **Latency/Cost** — что **именно** стало медленнее / съело квоту? (конкретная операция + почему: N+1, full-scan, лишний API-call). «Стало медленнее вообще» = не зачтено.
2. **Повторное использование** — на каком по счёту использовании раздражает и **чем именно**? (лишний шаг каждый раз / повторный ввод / шум в выводе).
3. **Edge case в данных** — назови **конкретный тип входа** который ломает: пустой файл? TODO-блок? non-ASCII / кириллица? отсутствующее поле? nested-структура? URL > лимита? «нестандартные данные» = не зачтено — нужен конкретный пример.

## Шаг 99.3 — Confidence Declaration

После Self-Lint — обязательный синтез. Для каждого свойства: confidence % **и** конкретный evidence.

**Формат evidence — двухсекционный «Учтено / Не учтено»** (явный gap layer per-property):

Каждая ячейка Evidence обязана содержать **обе** секции:

- **Учтено:** одно предложение **простым языком** — что именно покрыто / проверено / закрыто. Можно добавить в конце **техническую деталь предметно** (grep команда, file:line, имя индекса / mechanism).
- **Не учтено:** одно предложение **простым языком** — что НЕ покрыто И **почему** (выбор scope / эмпирика отсутствует / preemptive abstraction risk / acceptable trade-off / etc). Можно добавить в конце **техническую деталь** причины.

**Формат строки:** `**Учтено:** <простой текст> \| <тех. деталь>. **Не учтено:** <простой текст + причина> \| <тех. деталь>.`

**Примеры:**

✅ **Хорошо:**
> **Учтено:** не ломаем существующее: к Customer только добавляем связь, старые поля не трогаем. \| grep по routes (Шаг -1.3) — пусто. **Не учтено:** soft-delete flow (`deleted_at` cascade). \| → out of scope: требует отдельного [data] плана, acceptable risk per ROADMAP-019.

> **Учтено:** git fetch через with-secret.sh injection — токен не в stdout. **Не учтено:** detached HEAD edge case при pull. \| → войдёт в /code: проверка `git symbolic-ref HEAD` перед pull, graceful skip если detached.

❌ **Плохо:** «M:N semantics, CASCADE DELETE корректен» — жаргон без вывода и без gap'а.

❌ **Плохо:** «Учтено: всё проверил. Не учтено: ничего» — perfunctory без конкретики.

**Anti-perfunctory rule:** «Не учтено» секция **обязана называть конкретный gap с причиной**. «нет gap'ов» / «всё покрыто» = не зачтено — за каждым решением есть trade-off, нужно явно его назвать.

⛔ **Заполнить все 9 строк таблицы обязательно.** Неприменимые → `N/A — [обязательная причина]` (напр. `N/A — методологический текст, нет runtime`). Пропуск строки (не написать её вообще) = Шаг 99.3 не зачтён. Все 9 свойств **всегда видны** — это даёт пользователю полный контроль, что ни Performance ни Security не пропущены молча.

**Lite mode:** строки неприменимые к задаче (напр. "Масштабируемое" для однострочного багфикса) → пиши `N/A — Lite: [причина]` вместо %. Порог 80% применяется только к non-N/A строкам.

**Жёсткие правила:**
- ⛔ Evidence пустой → шаг не зачтён; заполни или вернись.
- ⛔ Confidence < 80% хотя бы по одному свойству → план не готов; вернись и закрой gap ИЛИ явно эскалируй в OPEN-QUESTIONS с рекомендацией.
- ⛔ **#4 (Без регрессий) — минимум 95%** (override общего 80% порога). Регрессии — асимметричный риск: cost поломки adjacent paths выше потенциальной пользы. <95% = блок до выполнения adjacent grep полностью ИЛИ явного N/A с обоснованием (для doc-only / methodology-rule / typo-fix задач где adjacent paths логически отсутствуют).
- ⛔ **Test isolation ≠ production reality.** Если confidence (особенно #5) опирается на test results — verify что тест-матрица включает PRODUCTION CONFIGURATION (combined mechanisms работающие одновременно), не только isolated mechanism testing. Изоляция → false high confidence; реальность раскрывается только при первом production-config запуске.
- ⛔ "100%" без конкретного evidence = самообман; снижай % или дополни evidence.
- ⛔ **Performance (8) / Security (9) — `N/A` только с обоснованием.** Если задача `[data]`/`[code]`/`[infra]` с runtime — Performance обязателен с реальным %, "и так быстро" без анализа = не зачтено. Если `[security]`/`[contract]`/`[data]` — Security обязателен с реальным %. Для методологии/doc/чистого текста — `N/A — [причина]` приемлемо, но строка ВСЕГДА видна (не пропускать).
- ⛔ **Evidence на жаргоне без вывода = не зачтено.** Каждая ячейка начинается с вывода обычным языком, потом детали. Чистый тех. жаргон ("M:N semantics, partial unique index L4/6") без человеческого вывода → вернуться и переписать.
- ⛔ **Двухсекционный формат обязателен.** Каждая non-N/A ячейка имеет **обе** секции: `Учтено:` + `Не учтено:`. Пропуск любой = не зачтено. «Не учтено: нет gap'ов» / «всё покрыто» = perfunctory, **не зачтено** — за каждым решением есть trade-off, нужно явно его назвать.
- ⛔ **«Не учтено» обязано содержать явный статус-тег** — пользователь должен понимать что с этим gap'ом произойдёт. Допустимые теги:

**Требование честности:**

**Требование доработки до 95%+ (stretch goal, не порог):**

Перед показом таблицы пользователю — **обязательная процедура доработки**:

1. **Анализ каждого non-N/A свойства:** для строк где confidence < 95% — спроси себя «что конкретно я могу проверить/уточнить чтобы поднять до 95%+?»
2. **Доработка:** выполни дополнительную проверку — grep, чтение файла, verification, фетч spec, integration check, thread-pull зависимости. **Реальное усиление evidence, не переписывание текста.**
3. **Пересчёт:** после доработки честно пересчитай confidence. Если поднялся до ≥95% — зафиксируй с новым evidence. Если остался <95% — оставь как gap и **явно заяви его** (формат: «Что мы не знаем по свойству N: [конкретно что нужно для повышения]»).

**Жёсткие правила (anti-inflation):**
- ⛔ **Inflated 95% без реальной доработки запрещён.** Если evidence не усилен (нет нового grep / нового чтения / новой verification) — оставить честный <95%. Поднимать % только когда есть **новый факт** в evidence ячейке.
- ⛔ **Stretch goal ≠ Risk Tier floor.** Floor (80%/95%/97% по Tier) — hard блок. Stretch (95%+) — soft target: «дораработай если возможно». Честный 85% выше floor 80% — ОК если попытка доработки сделана и зафиксирована как gap.
- ⛔ **Если после двух итераций доработки confidence остаётся <95%** — остановись. Не зацикливайся. Зафиксируй как «Что мы не знаем» с конкретикой что нужно (новые данные / runtime test / consumer feedback).
- ✅ **Цель:** план реально лучше к моменту показа пользователю, не «выглядит уверенно». Stretch goal — это процесс доработки, не отчётность.

**Класс который закрывает (G-046 lesson):** «агент игнорирует только что внедрённое им же правило — формально заполняет цифры без честного пересмотра». Stretch goal требует **действия** (новой проверки), не только перепродумывания % — это L3 регулятор (процедурный шаг), не L1 (промпт-правило которое можно проигнорировать).

## Шаг 99.54 — Draft Maps (Full mode, все типы задач)

> **Цель:** **UI visualization плана для пользователя** — схематично показать «какие изменения будут» как summary для понимания сути. Чисто read-only визуализация **зрелого плана** (после Pre-Mortem и Confidence Declaration). **Не** scope check tool — scope валидируется в Шагах -1.2 / -1.3 / 0.7 текстом. **Не** связан с контрактом ([contract]-задач Шага 1.7 — Mermaid и API contract это разные плоскости).
>
> **Почему здесь, не в начале команды:** генерация ПОСЛЕ зрелого плана точнее отражает финальное решение. Pre-Mortem (Шаг 98) мог изменить scope — ранний draft стал бы stale.

**Принцип:** ЛЮБОЙ план меняет что-то — нужно показать пользователю **визуально что станет после выполнения плана**. Draft Maps — это не "только если меняешь карту", это "визуализируй изменение, выбери подходящий тип". `N/A` — **редкое исключение** (чистый текст / typo, где визуализировать нечего), не default.

**Дерево решений — выбери тип по контексту плана (по порядку):**

1. 🗺 **USER-MAP?** — план меняет акторов / flows / UX / capabilities / пользовательские сценарии → draft USER-MAP
2. 🏗 **SYSTEM-MAP?** — план меняет компоненты / связи между сервисами / архитектурные границы / интеграции → draft SYSTEM-MAP
3. 📦 **ARTIFACT-MAP?** — план меняет артефакты / команды / read-write паттерны / nodes или edges → draft ARTIFACT-MAP
4. 🔀 **Ни один из трёх не подходит, но план что-то меняет** (логика внутри команды, правило, процесс, алгоритм) → **ad-hoc «было → станет»** flowchart (см. ниже)
5. **Нечего визуализировать** (чистый текст, typo, переформулировка одного предложения) → `N/A — нечего визуализировать: [причина]`

Несколько типов могут сработать одновременно (план меняет и акторов, и компоненты) → сгенерировать каждый затронутый.

**🔀 Ad-hoc «было → станет» draft** (вариант 4 — фолбэк когда ни одна постоянная карта не подходит):
- Graph direction: `graph LR`
- Nodes — ключевые элементы что меняются (напр. "7 свойств" → "9 свойств", "draft только 3 карты" → "draft всегда")
- Edges между Было→Станет с label характера изменения (`расширяется`, `сужается`, `заменяется`)
- ⛔ **Subgraph-имена НЕ могут быть endpoints стрелок** (closes G-085). `Было ==>|label| Станет` — невалидно: subgraph-имя не является node ID. Использовать **flat nodes без subgraph-контейнеров** для ad-hoc «Было→Станет»; разделение Было/Станет достигается позиционированием nodes, а не subgraph-блоками.
- **Preview-only:** ad-hoc draft НЕ сохраняется в файл, живёт только как ссылка в сообщении /plan. `/code` его не финализирует (нет постоянной карты для ad-hoc).
- Лимит ~12 nodes — это эскиз изменения, не полная схема.

**Draft = touched scope only:** только nodes/edges затронутые планом + минимальные anchor nodes для контекста. Не генерировать полную карту.

**🟪 Deferred-кластер (C — обязателен если в Confidence Declaration есть `→ deferred` / `→ out of scope`):**

В draft (любого типа, включая ad-hoc «было→станет») добавь отдельный subgraph-кластер с **серыми dashed-узлами** — что этот план осознанно **НЕ** делает. Источник — пункты «Не учтено → deferred / out of scope» из Шага 99.3 (НЕ весь проектный backlog — только этот план).

    D1["<пункт из 'Не учтено'><br/>→ out of scope: <причина>"]:::deferred
end
```

- Узлы — короткие (≤60 симв), по одному на пункт `→ deferred`/`→ out of scope`.
- Не связывать стрелками с основным графом (это «что НЕ делаем», не часть flow) — отдельный subgraph рядом.
- Если таких пунктов нет → кластер не добавлять.
- **Per-plan, эфемерно:** это отложенное **этого плана**. Проектный агрегат — отдельно, через `/scope-out` (anchor-узел в living maps). Не дублировать: draft-кластер = «что я отложил здесь», `/scope-out` = «что отложено в проекте».
