# /plan — Универсальная команда планирования

Ты — архитектурный критик проекта. Задача не просто составить план, а найти всё что может пойти не так — до того как написана первая строка кода.

**ЗАПРЕЩЕНО:** писать код, изменять файлы, создавать миграции. Только план, риски, вопросы и checklist.

---

## Рекомендуемая модель

**Strategy:** Default (Sonnet) — основной выбор. Upgrade to Capable (Opus) только при триггерах.

**Default tier (Sonnet):** Основной выбор для всех планов. Достаточна для архитектурного анализа, диагностики, синтеза решений.

**Upgrade to Capable tier (Opus) if:**
- `[contract]` + threat model сложнее стандартного
- `[security]` новый endpoint с нетривиальным threat-моделем
- Multi-service refactor (50+ файлов, много зависимостей)
- Обнаружена системная проблема в диагностике (class-bug в 3+ файлах)
- User explicitly asks for deeper analysis

**❌ Fast tier (Haiku):** НЕ рекомендуется для регулярных планов
- Риск: неправильный синтез, упущение консистентности
- Допускается только для grep/validation если у тебя явно нет reasoning-based шагов
- Лучше переплатить на Default чем пропустить архитектурное нарушение

**Mid-task escalation:** нет (анализ в /plan не требует mid-task переоценки как /code)

**Pre-flight model check:** **да — при старте команды** спроси пользователя какая модель активна (или используй ранее подтверждённую в сессии) и сравни с Default tier. Если mismatch ≥ 2 ступени — пауза + рекомендация перед началом анализа. См. `.claude/model-tiers.md` секция Pre-flight model check.

## Шаг -4: Correction check (первый, перед всем остальным)

Прочитай текст нового задания и **оцени семантически**: описывает ли оно ситуацию, где прошлая работа агента была неполной, неверной или что-то важное было пропущено?

**Это НЕ поиск ключевых слов.** Оценивай смысл и контекст:
- Задание содержит неявную или явную критику предыдущего решения?
- Указывает что что-то сделано "не так", "не полностью" или "не системно"?
- Ссылается на несделанное рядом с тем что уже сделано?
- Выражает неудовлетворённость результатом предыдущей итерации?

**Примеры коррекции** (оценивай смысл, а не наличие конкретных слов):
> "не нравится что только по этим словам определяем" — нет триггерных слов, но явная критика реализации
> "мне кажется системное решение не было реализовано" — недовольство результатом
> "ты добавил X, а Y — нет" — указание на пропуск
> "хочу пересмотреть как мы реализовали" — запрос на пересмотр

**Примеры НЕ-коррекции** (новые задачи и уточнения):
> "добавь функцию X" — новая задача без претензии к прошлому
> "как подойти к Y?" — открытый вопрос
> "расширь Z дополнительным поведением" — расширение без критики

Если ДА → **ПЕРЕД** pre-flight предложить:
```
📝 Задание содержит указание на пропуск/ошибку в прошлой работе.
   Зафиксировать в AGENT-GAPS.md? (y/n)
   Категория: [prompt-gap | context-gap | logic-gap | assumption-gap | completeness-gap | scope-gap]
   Гипотеза: [одна строка — почему агент это пропустил]
```

Если НЕТ → продолжить к Шагу -3.

Если `AGENT-GAPS.md` не существует в проекте → пропустить тихо.

## Шаг -3: Pre-flight checks (умный гейт)

Каждый триггер — **предложение с подтверждением**, никогда не авто-запуск.

### Подшаг 0 — Проверка незавершённого /code

1. Прочитать `.claude/state/triggers.json`, поле `last_plan_session`
2. Если `last_plan_session.code_run === false`:

```
⚠️ Предыдущий /plan не завершён /code:
   Task:    {task_id}
   Service: {service}
   Started: {completed_at}

Варианты:
  a) Запустить /code для прошлой задачи сначала (рекомендуется)
  b) Отметить как abandoned и продолжить новый /plan
  c) Игнорировать (инкремент skipped_warnings.code_skipped)
```

### Подшаг 1 — Чтение state и инкремент

1. Прочитать `.claude/state/triggers.json`
2. Определить контекст задачи (для multi-service проектов — какой сервис)
3. Инкрементировать счётчики:
   - `last_architecture_audit.plans_since += 1` (если архитектурные триггеры применимы)
   - `last_sync_vision.plans_since += 1` (если есть vision документы)
   - `last_retro.plans_since += 1`
   - `last_product_review.plans_since += 1`
   - `last_product_check.plans_since += 1`
   - `last_product_vision.plans_since += 1`
   - `last_user_map_sync.plans_since += 1` (если есть USER-MAP.md)
4. Сохранить triggers.json

### Подшаг 2 — Триггеры (порядок, по одному вопросу за раз)

Каждый вопрос **обязан** включать рекомендуемую модель для предложенной команды (см. `.claude/model-tiers.md` per-command матрица).

**Триггер: /architecture-audit** (если проект имеет архитектурную карту):
- `last_architecture_audit.plans_since` ≥ 5
- → "Архитектурный аудит давно не запускался (N=…). Запустить?
   Рекомендуемая модель: **Default tier** (Sonnet) — типично для audit.
   (y/n/skip)"

**Триггер: /sync-vision** (если есть vision документы):
- `last_sync_vision.plans_since` ≥ 5 И задача меняет контракты/архитектуру
- → "Меняются контракты. Запустить /sync-vision?
   Рекомендуемая модель: **Default tier** (Capable если 10+ inbox И 5+ OQ одновременно).
   (y/n/skip)"

**Триггер: /retro:**
- `last_retro.plans_since` ≥ 15 → "Накопилось N планов без ретроспективы. Запустить /retro?
   Рекомендуемая модель: **Default tier** (Capable если 60+ DEVLOG entries за период).
   (y/n/skip)"
- `last_retro.plans_since` ≥ 20 → то же сообщение, но skip требует явной причины:
   "Пропустить /retro (N планов). Причина: ___" → инкрементировать `skipped_warnings.retro`
- `skipped_warnings.retro` ≥ 3 → ⛔ HARD BLOCK: "Ретроспектива пропускалась N раз подряд.
   /retro обязателен перед следующим /plan. Запустить: (y)" — только y, нет skip.

**Триггер: /product-review** (если есть IDEAS.md):
- Записей в IDEAS.md без `[reviewed]` ≥ 7 ИЛИ `last_product_review.plans_since` ≥ 10
- → "Накопились сигналы (N unreviewed). Запустить /product-review?
   Рекомендуемая модель: **Default tier** (Fast если < 5 IDEAS, Capable если 20+).
   (y/n/skip)"

**Триггер: /product-check** (если есть PRODUCT.md):
- `last_product_check.plans_since` ≥ 5
- → "Изменено N файлов команд. Запустить /product-check?
   Рекомендуемая модель: **Fast tier** (Haiku) — структурное сравнение.
   (y/n/skip)"

**Триггер: USER-MAP sync** (если есть USER-MAP.md):
- `last_user_map_sync.plans_since` ≥ 10 И (добавлена новая команда ИЛИ изменился основной workflow)
- → "USER-MAP может требовать обновления (новая capability или workflow). Проверить и обновить?
   Рекомендуемая модель: **Fast tier** (Haiku) — review existing diagram.
   (y/n/skip)"

**Триггер: /product-vision:**
- `last_product_vision.plans_since` ≥ 30
- → "Стратегическая работа может быть устаревшей. Запустить /product-vision?
   Рекомендуемая модель: **Capable tier** (Opus) — strategic depth обязательна.
   (y/n/skip)"

**Триггер: новая идея без контекста:**
- Запрос описывает фичу не упомянутую в IDEAS.md/ROADMAP.md
- → "Добавить в IDEAS.md перед планированием? (y/n)"

### Подшаг 3 — Stale Open Questions (если есть OPEN-QUESTIONS.md)

Для каждого OQ в области текущей задачи:
- Если status=Open И возраст > 14 дней — показать
- Если истёк `Reminder.Expires` — ⛔ HARD BLOCK до решения PM

### Подшаг 4 — Hard blocks

- Новый сервис/компонент без /architecture-audit → ⛔ требует запуска
- EXPIRED OQ → ⛔ требует решения PM
- Иначе → продолжаем к Шагу -2

## Шаг -1 — Диагностика

**1. Симптом или причина?**
- Симптом → найди причину, исправь её
- Причина → определи класс проблемы

**2. Код или промпт?**
- Доменное ограничение → ок
- Кейс-ограничение → красный флаг

**3. N-й фикс одной проблемы?**
- grep DEVLOG по тегу за 7 дней
- Если N ≥ 2 → ⛔ СТОП → запустить `/diagnose` (не продолжать /plan без root cause)

**4. Точечное или структурное решение?** *(явно ответить — не пропускать)*
- **ТОЧЕЧНОЕ** (чинит symptom в одной точке) → обязательно: "что предотвращает рецидив?"
- **СТРУКТУРНОЕ** (чинит класс проблем) → переходи к Шагу 0.5 для Level 4+ анализа

## Шаг -1.2 — Осознанная полнота решения (Full mode only)

Только для **Full mode**. Lite mode этот шаг пропускает.

Каждое предложенное решение ДОЛЖНО явно указать:

**1. Что закрывается?** (main path, happy path)
- Примеры: "устраняет регрессию в основных сценариях", "фиксит root cause в 80% случаев"

**2. Что НЕ закрывается?** (gaps, edge cases, параллельные пути)
- Явный список: "edge case 1 не покрыт", "параллельный путь A остаётся", "асинхронный сценарий B не тестирован"

**3. Приемлем ли этот gap или требует action?**
- Если gap критичен → "нужны дополнительные шаги: [список]"
- Если gap некритичен → "OK потому что [обоснование]"

⛔ Без этого анализа → план не завершён. Вопрос: "Что мы не делаем и почему это OK?"

## Шаг -1.3 — Adjacent Impact Scan (Full mode only)

**Принцип:** назвать что ещё может быть затронуто ДО выбора решения — пока ещё можно скорректировать scope.

**1. Перечисли смежные зоны:**
Что читает компонент который меняешь? Что зависит от него?
- Для `[methodology]`: ARTIFACT-MAP, SYSTEM-MAP, USER-MAP, PRODUCT.md — всегда в списке; **если добавляется новый файл в `commands/` или `templates/` — попадает ли он к консьюмерам через `sync-methodology.sh`? Если нет — добавить в `templates/` или явно обосновать out-of-scope** (closes G-025)
- Для любого проекта — **Gitignore ownership check:** если целевой файл физически присутствует в дереве, проверить: `git check-ignore -v <path>`. Hit → файл принадлежит upstream repo (например, `.claude/commands/*.md` в консьюмере = копии из it-dev-methodology). Изменение = отдельный PR в upstream, не в текущем проекте. Если git недоступен → предупредить разработчика и проверить вручную. **Исключение:** build-артефакты (`dist/`, `*.pyc`, `node_modules/`) — не methodology ownership, редактировать нормально. (closes G-007)
- Для `[code]`: параллельные модули с тем же паттерном, shared state, event consumers
- Для `[data]`: downstream сервисы, кеши, consumer модели

**2. Для каждой смежной зоны ответь:**
- Затронута изменением? (y / n)
- Если **y** → включить в план ИЛИ явно назвать как out-of-scope с причиной

**3. Класс решения:**

| Тип | Признак | Обязательный вопрос |
|---|---|---|
| Точечный | Меняется ровно одна точка | "Что предотвращает рецидив в этой же точке?" |
| Структурный | Меняется паттерн или класс | "Все ли экземпляры паттерна найдены?" |
| Level 4+ | Архитектурный барьер | "Есть ли альтернативный путь который нужно закрыть?" |

⛔ Без явного ответа на пункт 3 → план неполный.

## Шаг 0.1 — Early pre-mortem (перед planning, не после)

**Принцип:** Real value of pre-mortem до начала planning, на этапе assumptions, не после того как план написан и люди attached к решениям.

Задай 3 вопроса ДО начала архитектурного анализа:

1. **Что мы предполагаем?**
   - Какие состояния внешних сервисов / платформ мы считаем true?
   - Примеры: "Gemini API всегда отвечает", "user всегда авторизован", "кеш актуален"
   - Напиши 3-5 assumptions

2. **Что может пойти не так в первый раз?**
   - Какой сценарий обрушит план на неделю 1?
   - Примеры: "API quota exceeded", "OS case sensitivity", "race condition при одновременных запросах"
   - Напиши 2-3 early failure modes

3. **Какое внешнее состояние может разрушить?**
   - Что я не контролирую и что изменится независимо от моего кода?
   - Примеры: "Gemini history limit", "git network", "database migration lock"
   - Напиши 2-3 external state hazards

**Если любое assumption неверно или не сформулировано** → ⛔ СТОП перед планированием. Переформулируй assumptions чтобы были проверяемы.

## Шаг 0.5 — Стратегический выбор инструмента

Перед "как реализовать" — спроси "правильный ли это класс решения?"

- Есть нативный примитив платформы? (Function Calling, schema, type system)
- Не строим ли keyword-матчер вместо использования LLM-native механизма?

**Уровни регулятора (для поведения агентов):**
1. Промпт-правило (игнорируется)
2. Description инструмента (слабо)
3. Few-shot примеры (средне)
4. Структура данных (сильно)
5. Отсутствие альтернативного пути (очень сильно)
6. Schema constraint (гарантия)

Для критичной фичи — уровень 4+ обязателен.

## Шаг 0.7 — Source Conflict Check

Запускается если задача интегрирует данные из inbox/чата/внешнего источника.

### Подшаг 0 — Tier 0 inventory (ОБЯЗАТЕЛЬНО для архитектурных задач)

**Принцип:** Перед валидацией внешних источников — убедись что прочитаны **внутренние принятые design specs**. Иначе риск переосмысления того что уже спроектировано (prior context-gap: agent предложил альтернативную архитектуру, не зная что internal design spec уже описывает принятую модель).

Просканировать в порядке приоритета:
- **Tier 0 (internal accepted specs):** `docs/services/*/`, `docs/architecture/`, `inbox/VCD*` — что уже принято командой
- **Tier 1 (external inputs):** `inbox/Competitor-*`, прочие inbox/ файлы — внешние сигналы
- **Tier 2 (background):** ADR archive, claude-knowledge/ если существует

**Output:** Список Tier 0 источников релевантных задаче (или "нет — задача не архитектурная"). ⛔ Если Tier 0 не отсканирован при `[code]`/`[contract]`/`[data]` → план не валиден.

## Шаг 99.3 — Confidence Declaration

После Self-Lint — обязательный синтез. Для каждого свойства: confidence % **и** конкретный evidence.

**Формат evidence — «вывод + детали»** (не технический жаргон без контекста):

**Жёсткие правила:**
- ⛔ Evidence пустой → шаг не зачтён; заполни или вернись.
- ⛔ Confidence < 80% хотя бы по одному свойству → план не готов; вернись и закрой gap ИЛИ явно эскалируй в OPEN-QUESTIONS с рекомендацией.
- ⛔ **#4 (Без регрессий) — минимум 95%** (override общего 80% порога). Регрессии — асимметричный риск: cost поломки adjacent paths выше потенциальной пользы. <95% = блок до выполнения adjacent grep полностью ИЛИ явного N/A с обоснованием (для doc-only / methodology-rule / typo-fix задач где adjacent paths логически отсутствуют).
- ⛔ **Test isolation ≠ production reality.** Если confidence (особенно #5) опирается на test results — verify что тест-матрица включает PRODUCTION CONFIGURATION (combined mechanisms работающие одновременно), не только isolated mechanism testing. Изоляция → false high confidence; реальность раскрывается только при первом production-config запуске.
- ⛔ "100%" без конкретного evidence = самообман; снижай % или дополни evidence.
- ⛔ **Conditional триггер сработал → `N/A` запрещён.** Если задача `[data]`/`[code]`/`[infra]` с runtime — Performance (8) обязателен с реальным %. Если `[security]`/`[contract]`/`[data]` — Security (9) обязателен. "И так быстро" / "и так безопасно" без анализа = не зачтено.
- ⛔ **Evidence на жаргоне без вывода = не зачтено.** Каждая ячейка начинается с вывода обычным языком, потом детали. Чистый тех. жаргон ("M:N semantics, partial unique index L4/6") без человеческого вывода → вернуться и переписать.
- 80-99% — норма, если evidence конкретен и риски записаны в Шаг 3 "Риски".

**Требование честности:**
- ⛔ Все non-N/A строки ≥90% **без объяснения** → подозрительно: назови явно почему нет слабых мест.
- Обязательно: в финальной таблице укажи МИНИМУМ одну строку с реальным gap, ограничением, или non-trivial риском — даже если confidence высокий. Если таковых нет — напиши одну строку "Чего мы не знаем: [конкретно]".
- Цель не "показать уверенность" — цель "найти что может пойти не так". Формальный 85% без self-challenge = сигнал недостаточной критичности.
