# /scan-sources — Сканирование внешних источников знаний

> **Цель:** единая точка входа для работы с реестром внешних источников (`external-sources.md`) — сканировать новый материал с YouTube / Telegram / repo-каналов, анализировать его research-логикой и захватывать decision-relevant выводы в feedback loop (`[research:X]` + IDEAS). Команда управляет реестром целиком: добавить / показать / убрать источник + сканировать.

**Отличие от `/research`:** `/research` исследует **один вопрос** ad-hoc. `/scan-sources` **обходит реестр** источников и ищет что нового с прошлого скана. Движок анализа общий — research-логика (WebFetch/WebSearch + verdict + DEVLOG-захват).

| | `/scan-sources` | `/research` |
|---|---|---|
| Вход | реестр `external-sources.md` (N источников) | один вопрос |
| Триггер скана | что нового с `last_scanned` | decision context |
| Output | `[research:X]` per-источник + IDEAS-сигналы | один `[research:X]` |

---

## Границы (что команда НЕ делает)

- ⛔ **Не автоматизирует мониторинг** (Граница 5 — не автоматизирует sync). Запуск только **ручной** пользователем. Нет cron, нет фонового слежения за каналами.
- ⛔ **Не runtime-система** (Граница 3). Команда не «подписывается» на каналы — она при ручном запуске делает WebFetch/WebSearch по URL из реестра.
- **Scope v1 — публичные URL.** WebFetch достаёт публичные страницы (repo-релизы, публичные YouTube/Telegram-каналы по URL). Приватные Telegram-каналы и YouTube-транскрипты за login-wall **недостижимы** — команда честно помечает такой источник `login-required`, не выдаёт пустоту за «ничего нового».

---

## Рекомендуемая модель

**Extended (UI settings):** effort: **High** · thinking: **ON** — cross-source synthesis «что нового decision-relevant» = reasoning. См. `.claude/model-tiers.md` § Effort & Thinking.

**Default tier:** **Default** (Sonnet) — WebFetch/WebSearch + synthesis + structured DEVLOG-захват достаточно.
**Upgrade to Capable if:** ≥3 источника с conflicting сигналами требуют cross-source reasoning; стратегический вывод с нетривиальным trade-off.
**Downgrade to Fast:** ❌ НЕ рекомендуется — synthesis «что полезно» требует reasoning.
**Mid-task escalation:** если при скане найдены conflicting сигналы между источниками → предложить Capable.
**Pre-flight model check:** да — спроси пользователя какая модель активна при старте (или используй подтверждённую в сессии).

## Шаг 0 — Резолв реестра

1. Прочитать `CLAUDE.local.md` → `doc_repo_path`.
   - `doc_repo_path` задан (two-repo) → реестр в `<doc_repo_path>/external-sources.md` (методологические источники) ИЛИ в root, смотря где файл существует. Методология: `external-sources.md` живёт и в code-repo root, и в doc-repo — читать тот что релевантен задаче (по умолчанию doc-repo, где живут DEVLOG/IDEAS-выводы).
   - `doc_repo_path: null` (single-repo) → реестр в root `external-sources.md`.
2. Файл **отсутствует** → graceful: «Реестр `external-sources.md` не найден. Создаётся через `sync-methodology.sh`. Запусти sync или добавь первый источник: `/scan-sources добавь <url>`.»
3. Файл **есть, но без строк-источников** (только заголовки/example) → «Реестр пуст. Добавь источник: `/scan-sources добавь <url>`.»

## Шаг 1 — Диспетч намерения (natural-language)

Команда — единая точка входа. Распознай намерение из текста запроса (не жёсткие флаги):

| Намерение пользователя (примеры) | Режим | Перейти к |
|---|---|---|
| `/scan-sources` (без аргументов), `просканируй`, `что нового` | **scan** (дефолт) | Шаг 2 |
| `добавь <url>`, `добавь источник <url> ловить X` | **add** | Шаг 3 |
| `покажи всё`, `список`, `что в реестре` | **list** | Шаг 4 |
| `убери <X>`, `удали источник N` | **remove** | Шаг 5 |
| `только <канал>`, `просканируй <url>` | **scan-one** | Шаг 2 (один источник) |

Неоднозначно → спроси одним вопросом какой режим.

## Шаг 2 — Scan (дефолт) / scan-one

**Для каждого источника** в реестре (scan-one — только указанный):

1. **Watermark:** прочитать колонку `last_scanned`. Сканировать если `last_scanned` пусто (`—`) ИЛИ старше сегодня. (scan-one игнорирует watermark — сканирует всегда.)
2. **Fetch:** выбрать метод по URL:
   - `file://`-URL → **Read локальный файл** (дайджест созданный внешним fetcher'ом). Файл не существует → статус `unreachable`.
   - Публичный URL → **WebFetch** (WebSearch если нужен более свежий индекс). Цель — найти материал, появившийся **после** `last_scanned`.
3. **Per-источник статус** (обязательно показать — закрывает RPN-252 «login-wall выдаёт пустоту за тишину»):
   - `fetched` — контент получен, проанализирован (WebFetch **или** `file://`-дайджест).
   - `unreachable` — URL не открылся (сеть / 404 / файл не найден) → `last_scanned` **НЕ** обновлять.
   - `login-required` — канал за login-wall (приватный TG / YouTube-транскрипт недоступен без fetcher'а) → `last_scanned` **НЕ** обновлять, сообщить что источник вне scope v1 или требует `/scan-sources-full`.
   - `nothing-new` — fetched, но нового материала с `last_scanned` нет.
4. **Анализ (research-логика):** для нового материала — что из этого **decision-relevant** для проекта? Не пересказ, а вывод с verdict (`viable` / `not-viable` / `confirmed` / `conditional` / `unclear`). Шум (не влияет на решение) — отбросить, не захватывать.
   - ⛔ **Большой источник → читать, не skim'ить заголовки** (closes coverage-gap): если дайджест/источник крупный (> ~50KB или > ~30 элементов) — **делегировать чтение exploration-subagent'у** (Agent tool), который читает **тело** материала по частям и возвращает структурированные находки + **coverage-log** (сколько элементов прочитано / отброшено / по каким категориям шума). Это ровно тот паттерн «exploration sub-agent» который методология рекомендует — не засоряет основной контекст. Agent tool недоступен → читать частями самому.
   - ⛔ **«Отброшено как шум» легитимно только если ПРОЧИТАНО**, не по заголовку. Title-скимминг вместо чтения = неполный анализ (no-silent-caps: что отброшено — назвать категориями с числом, не молча).
5. **Обновить `last_scanned`** = сегодняшняя дата (только для статусов `fetched` / `nothing-new`).

**Вывод скана (таблица статусов):**

```
🔍 Скан реестра (<N> источников):
| # | Источник | Статус | Новое decision-relevant |
|---|----------|--------|-------------------------|
| 1 | anthropic-cookbook | fetched | новый multi-agent pattern X → viable для skill-слоя |
| 2 | TG-канал Y | login-required | приватный — вне scope v1, пропущен |
| 3 | claude-code releases | nothing-new | — |
```

## Шаг 2.5 — Захват выводов (feedback loop)

Для каждого `decision-relevant` вывода — захватить, **иначе сигнал теряется** (QB6):

1. **`[research:X]` в DEVLOG** (история) — формат как в [/research](research.md) Шаг 4:
   ```
   [research:<slug>] → <что изучали (источник)>: <вывод>. <verdict>. Source: <url>
   ```
2. **Сигнал в IDEAS — по адресату** (куда влияет вывод):
   - Вывод про **методологию** (новый agent/hook/skill pattern, Claude Code API) → `IDEAS.md` методологии (doc-repo).
   - Вывод про **домен консьюмера** (рынок, конкурент, domain best-practice) → `IDEAS.md` **этого** консьюмера.
   - Адресат неочевиден → спросить одним вопросом.

Предложить пользователю (opt-out как в /research):
```
📋 Из скана — предлагаю захватить:
[research:<slug>] → <вывод> (DEVLOG)
IDEAS (<методология|консьюмер>): <сигнал одной строкой>

(y — записать всё / edit / skip)
```

При `y` — записать в DEVLOG (секция текущей даты) + IDEAS. Обновить `last_scanned` в реестре.

## Шаг 2.6 — Итоговый синтез (ОБЯЗАТЕЛЬНО, понятным языком)

> **Цель команды — не «отсканировать», а сделать вывод что нам полезно.** Per-source захват (Шаг 2.5) — это сырьё. Этот шаг сводит всё в **один вывод**. Без него скан = реестр находок, а не польза (closes coverage/synthesis-gap).

После анализа всех источников — вывести пользователю **единый синтез-блок**. Два жёстких требования:

**1. Понятный язык — обязателен.** Блок читает человек, не агент. ⛔ Запрещён жаргон в user-facing выводе: `Tier`, `verdict`, `viable/confirmed/conditional`, `harness`, `RPN`, `regulator level` и т.п. Пиши как объяснил бы коллеге: «что нашли», «почему это нам полезно», «что с этим делать». (Технические `verdict`-метки остаются **только** в DEVLOG-записи Шага 2.5 — это внутренний журнал, не этот блок.)

**2. Приоритизация — обязательна.** Не плоский список всех находок, а **ранжированный** «что полезнее»: сначала то что стоит сделать/добавить, потом подтверждения «мы и так это делаем», в конце — что отбросили и почему.

**Формат блока:**

```
## 🎯 Что из скана полезно нам

**Главное одной фразой:** <сквозная тема простыми словами>

**Стоит сделать / добавить (по убыванию пользы):**
1. <находка> — <почему полезно, простым языком> → <конкретный следующий шаг>
2. ...

**Подтверждения (мы это уже делаем правильно):**
- <находка> — <что именно подтвердилось>

**Отбросили как шум:** <категории + сколько> (читалось, не по заголовкам)
```

- Масштабируй под объём: 1-2 источника без новых сигналов → синтез 2-3 строки, без церемоний. Большой скан с находками → полный блок выше.
- Этот блок идёт **после** предложения захвата (Шаг 2.5) или вместе с ним — но всегда как финальный понятный вывод, который пользователь читает первым.

## Шаг 3 — Add (добавить источник)

1. Из запроса извлечь URL. Нет URL → спросить.
2. Спросить (с рекомендацией): **что ловить** (паттерны/релизы/идеи) + **частота** (`каждый /retro` / `ежеквартально` / `при релизе`). Рекомендация по умолчанию: «каждый /retro».
3. **Верифицировать URL** доступен (WebFetch HEAD-подобно) — если `login-required`/`unreachable`, предупредить что scope v1 это не покроет, но добавить можно (пользователь решает).
4. Вписать строку в `external-sources.md`: `| N | <name> | <url> | <что ловить> | <частота> | — |` (`last_scanned` = `—` для нового).
5. Подтвердить: показать добавленную строку целиком.

---

## Шаг 4 — List (показать реестр)

Прочитать `external-sources.md` → показать таблицу источников целиком (все колонки, включая `last_scanned`). Если реестр пуст → подсказать `добавь`.

---

## Шаг 5 — Remove (убрать источник)

1. Идентифицировать источник (по `#` или имени/URL из запроса).
2. Показать строку которую удаляем → **подтверждение** (y/n).
3. При `y` — удалить строку, перенумеровать `#`. DEVLOG `[fix:external-sources]` если источник был значимым.

---

## Связь с реестром и /retro

- `external-sources.md` — **источник правды** (отдельный версионируемый файл, QB9). `/scan-sources` — **интерфейс** к нему (читает/пишет). Команда не прячет данные «внутрь себя» — файл остаётся видимым.
- `/retro` Шаг 5.6 **пополняет** реестр кандидатами (на основе `[research:X]` тегов + AGENT-GAPS). `/scan-sources` **сканирует** реестр. Два разных триггера, один реестр — не дублируют.
- ⚠️ **Watchdog NB:** `agent-gaps-watchdog.py` (Stop hook) детектит research-находку только при **WebSearch** (не WebFetch-only). Если скан опирается на WebFetch — hook молчит, но это OK: команда сама пишет `[research:X]` (Шаг 2.5), watchdog — лишь страховка для incidental-находок вне команды.
