# /last-repo-changes — Что изменилось в repos workspace (простым языком)

> **Цель:** дать пользователю детальный отчёт **простым языком** по последним изменениям в repos workspace — что подтянулось/наработано, что ещё ждёт на remote, кто отстаёт — сгруппированный по смыслу, с **причинами** (из DEVLOG), без сырого git log.

**Read-only.** Команда НЕ делает pull и ничего не меняет — только читает (fetch refs + log + diff). Чтобы реально подтянуть изменения → `/pull`.

---

## Рекомендуемая модель

**Extended (UI settings):** effort: **Medium** · thinking: **ON** — синтез простым языком + группировка по смыслу + извлечение причин из DEVLOG = reasoning (не механика). См. `.claude/model-tiers.md` § Effort & Thinking.

**Default tier:** **Default** (Sonnet) — это аналитический синтез, не git-механика (в отличие от `/pull` который Fast·Low·OFF).
**Upgrade:** не требуется (объём ограничен сырым дампом коллектора).
**Pre-flight model check:** нет.

---

## Использование

```
/last-repo-changes                 — дельта последнего pull/merge по всем repos (дефолт)
/last-repo-changes 30              — последние 30 коммитов на текущей ветке каждого репо
/last-repo-changes "3 days ago"    — изменения за окно (любой git --since spec)
/last-repo-changes --no-fetch      — по кэшу, без fetch (быстро — сразу после /pull)
```

Аргумент окна и `--no-fetch` комбинируются: `/last-repo-changes 50 --no-fetch`.

## Шаг 1 — Сбор сырых данных

```bash
bash scripts/repo-changes-collect.sh [WINDOW] [--no-fetch]
```

Скрипт (read-only) для каждого репо workspace (кроме `it-dev-methodology`) выдаёт четыре блока:

- **DIVERGENCE** — ahead/behind каждой ветки vs upstream (картина «кто отстаёт»).
- **PULLED** — что подтянулось/наработано недавно на текущей ветке: полные commit-сообщения + **входящий diff DEVLOG** (источник правды о причинах — `[fix:X]`/`[feat:X]` с «Что/Почему»).
- **AVAILABLE** — что на remote, но ещё не локально (`HEAD..origin/<branch>`) + DEVLOG входящих.
- **STRUCTURAL** — diffstat с детекцией renames (перемещения/реструктуризация файлов).

Скрипт отсутствует (старая версия) → `⚠️ sync методологию до версии с repo-changes-collect.sh`.

## Шаг 2 — Синтез отчёта (простым языком)

Прочитай сырой дамп и **изложи простым языком, как объяснил бы коллеге** (Plain-language output rule). Структура отчёта:

**Для каждого репо где есть изменения:**
1. **Заголовок** — имя репо + одна строка «что это за репо» (код / документация / дистрибуция).
2. **Сгруппируй по смыслу, не по коммитам** — собери связанные коммиты в темы (реструктуризация / фикс класса X / новая фича Y / инфраструктура). Внутри темы — что и **почему** (причину бери из DEVLOG-блока, не угадывай из diffstat).
3. **Структурные перемещения** — если STRUCTURAL показал renames/moves — объясни смысл (что куда переехало и зачем), это часто важнее построчных правок.
4. **🆕 Ещё на remote** — если AVAILABLE непустой — назови что ждёт подтягивания (и подскажи `/pull`).

**Завершающий блок:**
- **🔀 Divergence** — какие ветки отстают/впереди (из DIVERGENCE). Отстающая `main` при работе в `ai-dev` — норма, скажи об этом, не алармируй.
- **🎯 Итог одной фразой** — что в сумме произошло across repos.

**Правила изложения:**
- Жаргон (теги `[fix:X]`, SHA, имена веток) — **вторичен**: в скобках или после простого вывода, не вместо него.
- Причина фикса — из DEVLOG. Если DEVLOG-блок пуст (код-репо без DEVLOG) — честно скажи «причина из commit-сообщения» и не выдумывай.
- Не пересказывай каждый коммит дословно — группируй и объясняй смысл.

## Чем отличается от соседних команд

- **`/pull`** — *механически* тянет изменения (ff-only). `/last-repo-changes` — *объясняет* что изменилось, ничего не тянет. Часто связка: `/pull` → `/last-repo-changes --no-fetch`.
- **`/pull-consumers`** (maintainer-only) — diff methodology-артефактов консьюмеров для `/retro`. Это про gaps консьюмеров, не про «что подтянулось в мой workspace».
