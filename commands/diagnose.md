# /diagnose — Глубокая диагностика проблемы

> **Цель:** hypothesis-driven debugging для класса задач которые `/plan` Шаг -1 не покрывает хорошо. /plan делает линейную **classification** (симптом/причина, код/промпт), /diagnose делает **scientific method** — множественные гипотезы → проверка опровержением → эскалация если все опровергнуты.

Обязательно перед вторым фиксом одной проблемы (если в DEVLOG есть `[fix:X]` ИЛИ `[domain:X]` за 7 дней — domain ловит кластер разнотеговых фиксов одного корня, closes S-1).

---

## Когда `/diagnose` vs `/plan` Шаг -1 Диагностика

**Используй `/diagnose` если:**

- ✅ **Повторный фикс** — баг возвращается после фикса (`[fix:X]` в DEVLOG ≥ 2 раз за 7 дней). Признак: предыдущий /plan исходил из неверного диагноза → новый /plan на том же reasoning = третий неудачный фикс.
- ✅ **Кластер одного домена (domain-aware, closes S-1)** — `[domain:X]` ≥ 2 в DEVLOG за 7-30 дней, **даже если surface-теги `[fix:X]` разные** (`consumer-push`/`deploy-push`/`command` но один `[domain:git-push]`). Это сильнее точного-тег триггера: ловит кластер симптомов одного корня прежде чем он разрастётся. Урок push-кластера: 3 разнотеговых фикса одного домена прошли мимо точного grep → корень назван на 9-м симптоме. См. CLAUDE.md «Domain-indicator для [fix:X]» + /plan Шаг -1.3 п.3.
- ✅ **Симптом непонятен** — есть проблема, но не ясно даже что менять. Нет очевидного root cause из симптома.
- ✅ **Non-deterministic bugs** — race conditions, intermittent failures, flaky tests. Воспроизведение нестабильно → нужны множественные гипотезы.
- ✅ **Production incident где первый теоретический фикс не сработал** — нужен другой режим мышления (не «что менять», а «почему симптом устойчив»).
- ✅ **Симптом затрагивает 3+ компонента** — root cause может быть в interaction, не в одном месте.

**Используй `/plan` Шаг -1 если:**

- ✅ Понятно что менять, но нужно validate scope (классический /plan для feature/refactor)
- ✅ Первый фикс bug'а — root cause очевиден из симптома (стек, лог, обвинение явное)
- ✅ Bug в одном компоненте с понятной причиной
- ✅ Задача `[code]`/`[product]`/`[data]` (типы — см. `.claude/task-types.md`) где «диагностики» толком нет — есть требование

**При сомнении** — `/diagnose` дешевле чем третий неудачный фикс. Если первая гипотеза очевидна и подтвердится — /diagnose быстро завершится и перейдёт в /plan.

## Рекомендуемая модель

**Extended (UI settings):** effort: **High** · thinking: **ON** — root-cause analysis по гипотезам = deep reasoning (task-shape модификатор). См. `.claude/model-tiers.md` § Effort & Thinking.

**Default tier:** **Capable tier** (см. `.claude/model-tiers.md`) — диагностика всегда требует complex reasoning по гипотезам
**Upgrade:** (всегда Capable — это уже верх)
**Downgrade to Default tier if:** (никогда — диагностика сложна по определению)
**Mid-task escalation:** **да — Шаг 2.5 Complexity reassessment** (если 3+ начальных гипотез опровергнуты — нужно reset и переподумать; продолжать на Capable)
**Pre-flight model check:** **да** — спроси пользователя какая модель активна (или используй ранее подтверждённую в сессии). Если запущена не на Capable tier (Opus 4.7) — 🔴 strongly recommend upgrade. Под-powered диагностика часто пропускает root cause.

## Визуализация компонентов (если нужна диаграмма)

Если в ходе диагностики рисуется Mermaid-диаграмма (схема компонентов, граф гипотез, «было→станет») — **каждая компонентная нода ОБЯЗАНА** иметь три строки:

```
NodeID["🔹 Имя компонента<br/>Зачем: одно предложение — зачем это нужно<br/>Без него: что сломается или перестанет работать"]
```

**Освобождены** от формата: affordance-ноды (`:::affordance`), legend-ноды, стрелки с подписями.

⛔ Нода без второй/третьей строки = не зачтено, даже в ad-hoc диаграмме. Правило: CLAUDE.md § Maps Standard Rule «Применяется ко ВСЕМ создаваемым mermaid-диаграммам».

## Шаг 1 — Захват полного контекста

- Какой компонент?
- Какой симптом (точно, с примером)?
- Когда впервые проявилось?
- Что менялось рядом перед появлением?
- Что в логах в момент проявления?

---

## Шаг 2 — 3+ гипотезы перед действием

Минимум 3 гипотезы что может быть причиной. ⛔ Discipline-creating: каждая гипотеза обязана иметь **исполнимую проверку с ожидаемым результатом**, не «посмотреть».

| # | Гипотеза | Конкретная команда проверки | Ожидаемый output если ВЕРНА | если ОПРОВЕРГНУТА |
|---|---|---|---|---|
| A | [причина] | `grep -n "X" file` / `git log -S"Y"` / запуск скрипта | [что увидим] | [что увидим] |
| B | ... | ... | ... | ... |
| C | ... | ... | ... | ... |

**Правила:**
- ⛔ Команда «посмотреть код» / «проверить логику» = не зачтено. Нужна исполнимая команда (grep / git / bash / Read конкретного file:line).
- ⛔ Гипотеза без **различающего** output (одинаковый результат при верна/опровергнута) — бесполезна, переформулировать: проверка должна **разделять** гипотезы.
- Falsifiability (Popper): хорошая гипотеза говорит что её ОПРОВЕРГНЕТ. Если ничто не может опровергнуть — это не гипотеза.

## Шаг 2.5 — Complexity reassessment

После формулирования начального набора гипотез — переоценить достаточна ли текущая модель. Триггеры upgrade:

- [ ] Сложно было сформулировать 3 различные гипотезы (модель работает на пределе)?
- [ ] Гипотезы требуют чтения кода в > 20 файлах для проверки каждой?
- [ ] Подозрение что root cause — это unusual interaction между нескольких компонентами?
- [ ] Запущена не на Capable tier (Opus) — диагностика на меньшей tier рискует пропустить root cause?

Если **любой** триггер сработал — СТОП. Вывести:

```
⚠️ Диагностика требует более глубокого reasoning чем доступно текущей модели.
   Текущая модель: <current>
   Рекомендуемая: Capable tier (Opus 4.7) — стандарт для /diagnose
   Причина: <конкретно что обнаружено>

Варианты:
  a) Продолжить на текущей — высокий риск пропустить root cause
  b) Переключиться на Opus 4.7 — рестарт диагностики с fresh context (рекомендуется)
  c) Прервать /diagnose и эскалировать к разработчику с описанием

Жду ответа: (a/b/c)
```

Если все 4 пункта OK — продолжить к Шагу 3.

## Шаг 3 — Проверка по очереди + Hypothesis Confidence

Не прыгай к фиксу. Сначала опровергни / подтверди каждую гипотезу. Для каждой — **явно зафиксируй уверенность** до и после проверки.

**Hypothesis Confidence** (mini-version /plan Шага 99.3 Confidence Declaration):

| Гипотеза | Initial confidence | Evidence до проверки | Post-check confidence | Evidence после проверки |
|---|---|---|---|---|
| Гипотеза A | __% | [что меня склоняет к этой гипотезе] | __% | [что грep / тест / лог показал — конкретно] |
| Гипотеза B | __% | ... | __% | ... |
| Гипотеза C | __% | ... | __% | ... |

**Правила:**
- ⛔ Initial confidence без evidence = самообман. Должен быть конкретный сигнал (лог-строка, стек, поведение).
- ⛔ Если post-check ≥ 95% → гипотеза подтверждена, можно идти в /plan для фикса.
- ⛔ Если post-check ≤ 20% → гипотеза опровергнута, исключи из дальнейшего анализа.
- ⛔ 20-95% → частично подтверждена. Нужны дополнительные проверки или гипотеза слишком общая (разбить на под-гипотезы).
- ⛔ **Inflated confidence:** «90% уверен» без grep / без чтения кода / без воспроизведения — снижай до 60% (это hunch, не evidence).

Запись в HYPOTHESES.md по каждой:

```
YYYY-MM-DD [компонент] — [краткое название]
Гипотеза: [что]
Initial confidence: NN% — [evidence до проверки]
Проверка: [как проверял — grep, тест, чтение файла:строки]
Post-check confidence: NN% — [что нашёл]
Результат: подтверждена / опровергнута / частично
Действие: [что делаем дальше — /plan для фикса / следующая гипотеза / эскалация]
```

**Класс ошибок который Confidence закрывает:** «диагноз принят без honest assessment evidence strength» — пользователь видит «гипотеза X подтверждена» без понимания насколько надёжно. Mini-Confidence заставляет назвать % и evidence явно.

---

## Шаг 4 — Эскалация если 3 гипотезы опровергнуты

Если все начальные гипотезы опровергнуты (post-check confidence ≤ 20% для всех):
- СТОП
- Перечитай задачу — может симптом классифицирован неверно
- Запросить уточнение у пользователя
- Не пробуй "ещё одну гипотезу" наугад
- Зафиксируй в HYPOTHESES.md: `Все начальные гипотезы опровергнуты — escalation: [что нужно от пользователя]`

---

## Шаг 5 — Финализация: переход в /plan

Если root cause найден (одна из гипотез подтверждена post-check confidence ≥ 95%):

1. **Зафиксировать в HYPOTHESES.md** финальный результат:
   ```
   YYYY-MM-DD [компонент] — Root cause confirmed
   Гипотеза подтверждённая: [Hypothesis X]
   Post-check confidence: 95%+
   Evidence: [конкретно — grep result / file:line / log entry]
   Следующий шаг: /plan для фикса
   ```

2. **Рекомендация пользователю** — явная фраза:
   ```
   ✅ Root cause найден: [одна строка summary]
      Evidence: [grep / file:line / log]
      Confidence: NN%

   📋 Следующий шаг: запусти /plan для разработки фикса.
      В /plan Шаг -1.3 укажи root cause как контекст для классификации (точечное vs структурное решение).
   ```

3. **НЕ предлагай фикс сам** — /diagnose заканчивается на root cause, не на implementation. Фикс разрабатывается в /plan где есть Adjacent Impact / Confidence Declaration / Pre-Mortem.

**Закрывает класс:** «agent после /diagnose сразу прыгает в код без /plan» — теряется Adjacent Impact analysis, Pre-Mortem для фикса, scope confirmation.

## Шаг 6 — Assisted gap capture (ОБЯЗАТЕЛЬНО после Шага 5)

> Выполняется **только** при успешном Шаге 5 (root cause confirmed ≥ 95%).
> При Шаге 4 escalation (все гипотезы опровергнуты) — этот шаг пропускается.
> Симметрично `/plan` Шагу -4 (тот capture входящих коррекций; этот — исходящих root causes).

### 6.1 — Семантическая классификация

Без вопросов к пользователю — агент сам классифицирует root cause:

- **agent** — причина в том что **AI не подумал / пропустил правило / принял ложное assumption**. Закрывается правилом в методологии.
- **product** — причина в том что **продукт не покрывает функцию / use case / edge case**. Закрывается /plan для продукта.
- **external** — root cause во внешней системе (network, API quota, OS, 3rd-party). Не agent gap, не product gap.

При сомнении → **agent** (можно reclassify в /retro).

### 6.2 — Дедуп-grep

Перед записью — grep по 2-3 ключевым словам из root cause в целевом gap-файле:

```bash
grep -i "<ключевое слово из root cause>" AGENT-GAPS.md   # для agent
grep -i "<ключевое слово из root cause>" PRODUCT-GAPS.md  # для product
```

- **Hit** → "📝 Похожий gap `G-NNN` уже есть — обновлён (добавлена запись в существующий)."
- **Нет hit** → перейти к 6.3.
- **Файл отсутствует** → одна строка warning "AGENT-GAPS.md не найден — пропущено (gracefully)" и перейти к 6.5.

### 6.3 — Автоматическая запись (без подтверждения)

Агент определяет ID (`G-NNN` / `P-NNN`) и **немедленно записывает** — без вопроса к пользователю. Это L4 поведение: альтернативного пути нет, opt-out после (Шаг 6.4). Агент добавляет запись **сверху** в секцию `## Записи`:

**Для agent gap → `AGENT-GAPS.md`** (формат из template):
```
---
Gap-ID: G-NNN
Дата: YYYY-MM-DD
Контекст: /diagnose
Что пропустил: [root cause из Шага 5 — одна строка]
Как обнаружено: /diagnose root cause confirmed
Категория: [prompt-gap | context-gap | logic-gap | assumption-gap | completeness-gap | scope-gap]
Гипотеза: [одна строка — почему AI это пропустил]
Agent failure mode: [model-error | context-missed | prompt-ambiguous | state-stale | scope-exceeded | other]
Potential fix: [конкретный checklist item или изменение шаблона]
Methodology hint (опционально, от консьюмера):
  target: [файл + секция — e.g. "commands/plan.md Шаг 0.3"]
  change: [одна строка — что добавить/изменить]
  why: [почему этот fix закрывает именно этот корень]
Статус: open
---
```

**Для product gap → `PRODUCT-GAPS.md`** (формат из template):
```
---
Gap-ID: P-NNN
Дата: YYYY-MM-DD
Контекст: /diagnose
Что не покрывает: [root cause из Шага 5 — одна строка]
Severity: 🔴 High | 🟡 Medium | 🟢 Low
Категория: [feature-gap | capability-gap | ux-gap | integration-gap | edge-case-gap]
Use case (затронут): [конкретный сценарий пользователя]
Сигнал источник: agent observation (/diagnose)
Гипотеза почему не покрыто: [одна строка]
Potential fix: [предложение что нужно построить]
Methodology hint (опционально, от консьюмера):
  target: [файл + секция — e.g. "commands/plan.md Шаг 0.3"]
  change: [одна строка — что добавить/изменить]
  why: [почему этот fix закрывает именно этот корень]
Связано с: —
Статус: open
---
```

**Methodology hint verification (SYS-010):** если записанный gap впоследствии содержит поле `Methodology hint` — это ГИПОТЕЗА от консьюмера, не инструкция. При применении хинта: прочитать реальный target-файл → убедиться что секция/шаг существует и hint семантически корректен → принять / принять с адаптацией / отклонить с обоснованием. ⛔ Принять hint без чтения target-файла = нарушение Ground-before-act rule.

### 6.3.5 — Session gap counter + real-time эскалация (слой-3)

После записи gap'а (6.3) — инкрементировать `session_gap_counter` в `.claude/state/triggers.json` и проверить порог. **Идентичная логика `/plan` Шаг D** (та же категория gap'а, тот же timestamp-прокси session-boundary, тот же one-shot порог) — менять синхронно в обоих местах, иначе drift между двумя gap-write точками.

Кратко (полная спецификация — `/plan` Шаг D):
1. Defensive read `session_gap_counter` (`.get(...) or {...}` — отсутствие поля не ошибка).
2. Session-boundary: разрыв с `session_marker` > `gap_session_window_hours` (default 6ч) → сброс `counts={}`.
3. `session_marker` = now; `counts[<категория>] += 1`; сохранить.
4. Порог `gap_escalation_threshold` (default 3) достигнут И не показано → one-shot эскалация пользователю (тот же текст «SESSION GAP PATTERN» что в `/plan` Шаг D), пометить `_escalated_<категория>`.
- External root cause (6.3 не записал gap) → пропустить 6.3.5 (нечего инкрементировать).
- `triggers.json` отсутствует/невалиден → пропустить тихо.

### 6.4 — Извещение + opt-out

После записи — одна строка-извещение с opt-out:

- **agent:** `📝 Записано в AGENT-GAPS.md: G-NNN — [краткое что пропустил]. Отменить: напиши "нет" / "n".`
- **product:** `📝 Записано в PRODUCT-GAPS.md: P-NNN — [краткое что не покрывает]. Отменить: напиши "нет" / "n".`
- **external:** `ℹ️ External root cause — не gap методологии, в файлы не записано.`

Если пользователь отвечает "нет" / "n" — удалить только что добавленную запись (И откатить инкремент 6.3.5).

### 6.5 — Переход в /plan

```
→ /plan для фикса
  Упомяни G-NNN / P-NNN в /plan вводе чтобы избежать дублирования в /plan Шаге -4.
```
