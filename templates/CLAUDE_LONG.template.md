## Сила регуляторов поведения (Level-4 framework) — расширенно

От слабого к сильному:

1. **Правило в системном промпте / MISSION** — игнорируется при достаточном контексте.
   - Why слабо: LLM имеет ограниченное внимание, длинные правила теряются.
   - Когда применять: когда нет более сильного механизма; как backup для прочих уровней.

2. **Description инструмента** — учитывается, но слабо.
   - Why: description видит модель при выборе инструмента, но не при использовании.
   - Когда применять: для guiding селекции инструмента.

3. **Few-shot примеры в промпте** — средняя сила, дрейфуют со временем.
   - Why: модель учится по примерам, но новый context может перевесить few-shot.
   - Когда применять: для демонстрации формата output.

4. **Структура входных данных** (что физически видит агент) — сильно.
   - Why: модель буквально видит только эти данные, контекст ограничен.
   - Когда применять: для критических ограничений.

5. **Отсутствие альтернативного пути** (один инструмент для задачи) — очень сильно.
   - Why: модель не может "забыть" использовать единственный инструмент.
   - Когда применять: для канонизации workflow.

6. **Function Calling schema / type constraint** (required параметры) — гарантия.
   - Why: системно невозможно вызвать без required.
   - Когда применять: для invariants нарушение которых ломает всё.

**Правило:** при добавлении нового поведения — начинать с уровня 4-6, не с уровня 1-3. Промпт-правило как первое решение = 🔵 Recommendation в review.

**Историческая мотивация:** этот framework возник из real-world проекта (2026-04-24) — MISSION-правила не держались под давлением коррекции. Гипотеза 1 (усилить текст) опровергнута — нарушения повторились в той же сессии. Гипотеза 2 (структурный регулятор CORRECTION_PROTOCOL) — подтверждена. Урок: при следующем нарушении того же типа — не добавлять правила, проверять уровень регулятора.

## Правило: level-4 проверка перед методологическими правилами (с обоснованием)

Перед принятием ЛЮБОГО методологического правила (новый lint, pre-mortem пункт, checklist-пункт, правило в `/review` или `/deploy`) — обязательно спросить:

> "Есть ли level-4+ структурный фикс той же проблемы?"

- **Level 4+:** schema constraint, type system, отсутствие альтернативного пути, cap на длину данных — срабатывает автоматически.
- **Level 1-3:** правило в команде, lint-чек, prompt-инструкция — требует дисциплины и памяти.

Если level-4 фикс существует — он primary, правила secondary.
Если level-4 невозможен — правила приемлемы как компромисс, но это надо явно зафиксировать.

⛔ **Запрещено:** принимать ТОЛЬКО level 1-3 правила без явного обоснования почему level-4 невозможен.

Симптом нарушения: добавили 3+ checklist-пункта на одну тему, через месяц тот же класс багов повторяется — правила не работают, нужен code-level фикс.

**Реальный пример из истории методологии:** при добавлении `Pre-flight model check` в Phase G1, выбор был между (a) правило в начале каждой команды (level-1), (b) централизованный реестр `model-tiers.md` + структурная секция в каждой команде (level-4). Выбран level-4 — невозможно "забыть упомянуть модель", потому что секция часть template.

## Model tier rule (расширенно)

Каждая команда методологии обязана содержать секцию `## Рекомендуемая модель` с 5 полями: Default tier / Upgrade to Capable / Downgrade to Fast / Mid-task escalation / Pre-flight model check.

Каноническая матрица — в [.claude/model-tiers.md](.claude/model-tiers.md) (методологический источник: [templates/model-tiers.md](templates/model-tiers.md)).

**Почему tier-абстракция, а не имена моделей:**
- Anthropic переименовывает модели каждые 6-12 месяцев (Sonnet 4.6 → 4.7 → 5.0 → ...).
- Если команды ссылаются на "claude-sonnet-4-6" — каждый ребренд требует правки 12 файлов.
- Tier-абстракция (Fast/Default/Extended/Capable) — единый mapping в одной таблице (`Mapping current model names → tiers`). Ребренд = правка одной строки.

**Почему Pre-flight model check обязателен:**
- Без него агент может молча работать на over- или under-powered модели — cost waste или quality risk.
- Pre-flight — это **level-5 регулятор** (отсутствие альтернативного пути): команда обязана выполнить check, нет варианта "пропустить".
- Это **не блокирующий** механизм — пользователь принимает финальное решение. Цель — guidance, не enforcement.

**Что считается mismatch:**
- ≥ 2 ступени разницы (например Capable на /product-check — над-powered by 2; Fast на /diagnose — под-powered by 2).
- 1 ступень — neutral, не fires (это нормальный headroom).

## Реализация через /code (расширенно)

После `/plan` и его подтверждения — реализация **обязательно** через `/code`. Прямая реализация запрещена.

**Причина:** `/code` обновляет `triggers.json` (`last_plan_session.code_run = true`), без чего следующий `/plan` покажет warning. Если разработчик говорит "просто реализуй" — Claude обязан спросить: "Запустить `/code` или запустить команду самому?".

**Что произойдёт если игнорировать:** триггер-механизм методологии перестаёт работать. `/plan` не знает что предыдущий план был реализован, продолжает показывать "незавершённый /code" warning. Дисциплина деградирует.

**Когда исключение разрешено:** микро-правка (опечатка в комментарии, форматирование) — допустимо без /code, но запись в DEVLOG с тегом `[micro]` обязательна. Грейлайн между микро и нет — субъективно; при сомнении — через /code.

## Deploy rule (расширенно)

Перед каждым деплоем (`<команда деплоя>`) — обязательно:
1. Запустить `/review` (если не запускался в этой сессии).

## Architecture decision rule (расширенно)

Перед любым архитектурным решением (новый модуль, изменение потока данных, новый сервис, новая интеграция):
1. Запустить subagent `architect` (`.claude/agents/architect.md`).
2. Claude **всегда** даёт свою рекомендацию ПЕРЕД запуском архитектора — чтобы получить независимое второе мнение, а не подтверждение.
3. Architect предлагает альтернативы — финальное решение принимает владелец.

**Почему сначала рекомендация Claude, потом architect:** иначе architect — это просто rubber-stamp того что Claude уже решил. Independent review требует чтобы было что ревьюить независимо.

## Fix rule (расширенно)

Перед любым фиксом (код или промпт):

### Симптом или причина?

- **Симптом** → найди причину, исправь её.
- **Причина** → определи класс проблемы → реализуй на правильном уровне абстракции (см. Level-4 framework).

**Почему различение важно:** фикс симптома работает в одном кейсе но не закрывает класс. Через 2 недели тот же класс проявляется в другом контексте — повторный фикс. Через 6 раз — крутимся по кругу.

### Локальный или системный баг?

- **Локальный** → точечный фикс, обосновать почему не может повториться.
- **Системный** → архитектурный фикс предпочтительнее (декоратор, middleware, schema constraint).

⛔ "Локальный" без обоснования — красный флаг, переходи к системному.

**Реальный пример:** real-world проект 2026-05-15 — специальный символ ломал обработку в двух модулях. Первый фикс (2026-05-12) — точечный, не нашёл второй экземпляр. Второй фикс понадобился через 3 дня — потому что не было grep'а по всему проекту. После этого class-bug-rule в /review стал обязательным.

## Capture продуктовых сигналов (расширенно)

В любом диалоге (не только `/plan`) — фиксировать сигнал в `IDEAS.md` **молча** (без упоминания пользователю) когда:

- Пользователь говорит "было бы удобно если...", "почему он не...", "я вынужден каждый раз..."
- Пользователь выражает удивление: "не знал что...", "я не видел..."
- Пользователь повторяет одно и то же действие вручную ≥3 раз в диалоге
- Текст интерфейса вызвал вопрос или непонимание

Не прерывать ответ — сначала ответить на запрос, запись добавить параллельно.

**Симптом нарушения:** IDEAS.md пуст после 10+ диалогов — сигналы были, но не зафиксированы.

**Почему молча:** упоминание "я записал в IDEAS" перегружает пользователя процессом. Цель — invisible signal capture, видимый только при /product-review.

**Что НЕ капчерить:** конкретные баги (это DEVLOG), идеи разработчика без сигнала от пользователя, технические задачи. См. таксономию в IDEAS.md.

## Принцип: security theater недопустим (расширенно)

Любая security-мера должна иметь:
1. Конкретную угрозу из списка выше против которой работает.
2. Способ проверить что мера реально работает (тест, не "установлена").
3. Понимание остаточных рисков.

### Симптомы security theater

- "Установить X" без конфигурации под конкретную угрозу.
- "Закрыть порт Y" без понимания через что реально атакуют.
- Решение от воображаемой угрозы при наличии реальных непокрытых.
- `default deny outgoing` без полного списка зависимостей приложения.

**Почему theater хуже отсутствия мер:** даёт false sense of security. Команда расслабляется ("у нас есть X"), реальные угрозы не закрываются. При инциденте обнаруживается что X не помогает.
