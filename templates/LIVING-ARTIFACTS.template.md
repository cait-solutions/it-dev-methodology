## Реестр живых артефактов

| Артефакт | Тип | Trigger обновления | Кто обновляет | Detection | Связанные артефакты |
|---|---|---|---|---|---|
| `LIVING-ARTIFACTS.md` | registry | добавление механизма; /plan Шаг 97 | Developer | validate-lar.sh (если есть) | ARTIFACT-MAP.md |

> **Инструкция по заполнению:**
>
> **Тип:** `hook` · `script` · `command` · `map` · `config` · `registry` · `doc` · `template` · `counter`
>
> **Trigger:** событие которое делает артефакт устаревшим. Конкретно: "добавлена новая команда" / "изменена схема triggers.json" / "каждый Full /plan". НЕ: "по мере необходимости".
>
> **Кто обновляет:** `Developer` / `sync-methodology.sh` / `hook (автоматически)` / `пользователь`
>
> **Detection:** как обнаружить что артефакт устарел/мёртв. Конкретно: `validate-X.sh` / `grep patterns` / `liveness check` / "нет детекта — gap!". НЕ: "визуально".
>
> **Связанные артефакты:** что ещё нужно обновить когда этот файл меняется. Основа для Adjacent Impact check в /plan Шаг -1.3.

## Refresh Policy

**Когда обновлять:**
- При каждом Full `/plan` → после Шаг 97 Sustainment Declaration → добавить строку для нового механизма
- При каждом `/code` Шаг 5 → обновить строку если lifecycle изменился
- При `/retro` → проверить есть ли артефакты без Detection (gap)

**Когда НЕ обновлять:**
- Typo/refactoring без изменения lifecycle
- Изменение контента артефакта без изменения trigger/detection/owner
