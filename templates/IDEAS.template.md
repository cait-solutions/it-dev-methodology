# IDEAS — {{Project Name}}

Сырые продуктовые сигналы из диалогов с пользователем и наблюдений за использованием.

Источник: автокапчер из диалогов с агентом (см. правило в CLAUDE.md) + ретроспективный анализ.

---

## Когда добавлять запись

- Пользователь сказал "было бы удобно если..."
- Пользователь сказал "я не знал что..."
- Пользователь сказал "почему он не..."
- Пользователь повторно делает одно и то же вручную (паттерн ≥ 3 раз)
- Пользователь выразил замешательство, удивление или непонимание
- Текст интерфейса вызвал вопрос или неоднозначность
- Пользователь описал обходной путь как норму

## Когда НЕ добавлять

- Конкретные баги → в DEVLOG с тегом `[fix:X]`
- Идеи разработчика без сигнала от пользователя
- Архитектурные решения → в `/plan` → ADR
- Жалобы без конкретного контекста ("работает плохо")

## Таксономия типов сигналов

| Тип | Когда использовать | Пример сигнала |
|---|---|---|
| `friction` | Лишние шаги, обход ограничения, ручная работа которая могла бы быть автоматической | "запускаю лот за лотом вручную" |
| `visibility-gap` | Пользователь не видит статус / состояние системы | "не знаю, проиндексировались ли все заметки" |
| `discovery-gap` | Пользователь не знал о существующей возможности | "не знал что так можно" |
| `ux-wording` | Текст UI вводит в заблуждение или не соответствует реальности | "написано 'обновляю', но это уже фон" |
| `context-confusion` | Пользователь перепутал режим / команду / контекст | "написал /cmd как заметку, не как команду" |
| `workflow-gap` | Отсутствует связь между двумя действиями, приходится вручную соединять | "запускаю /index, потом руками /sync" |
| `meta` | Сигнал о процессе разработки или о поведении агента | "не знаю какие возможности у меня есть" |

Используй один тип на запись. Если сигнал многоплановый — выбери доминирующий.

---

## Шаблон записи

```
## YYYY-MM-DD
- **Контекст:** после какой команды / диалога / действия
- **Сигнал:** что именно сказал или сделал пользователь (цитата или паттерн)
- **Тип:** friction | visibility-gap | discovery-gap | ux-wording | context-confusion | workflow-gap | meta
- **Почему важно:** почему это не разовое (паттерн, частота, последствия)
```

## Обработка сигналов

`/product-review` (раз в 14 дней или при ≥7 unreviewed) разбирает накопленные сигналы:

1. Группирует по типам и компонентам.
2. Ищет паттерны (одна команда в IDEAS ≥ 3 раз — высокий приоритет).
3. Предлагает 5-7 конкретных продуктовых решений с привязкой к данным.
4. High-приоритетные → предложение в ROADMAP.md → Considered (с явным "да" от владельца).

После разбора пометить запись `[reviewed YYYY-MM-DD]`.

---

## Пример записи

```
## 2026-04-30
- **Контекст:** /add сохраняет заметку в Inbox, но после git pull на рабочей машине заметки нет
- **Сигнал:** "проверь почему при пуше с гита в обсидиан не подтянулись сохранённые в боте заметки — ничего нет"
- **Тип:** visibility-gap
- **Почему важно:** пользователь не имел индикатора о причине: git push не вызывался вообще в /add. Vault на VPS ≠ vault на рабочей машине без git. Каждый раз требуется дополнительная диагностика.
```
