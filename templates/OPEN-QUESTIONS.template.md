# Open Questions — {{Project Name}}

Лог расхождений между vision/ADR и реальностью кода, или новой информацией, требующих решения PM.

> **ЗАПРЕЩЕНО** редактировать автоматически агентами. Только через явный процесс `/plan` или `/sync-vision`.

---

## Когда создаётся OQ-запись

- `/plan` Шаг 0.7 (Source Conflict Check) → Type C или Type E конфликт.
- `/sync-vision` → расхождение между vision/ADR и кодом.
- Архитектурный вопрос требующий решения владельца — не может быть решён внутри одного плана.

## Что НЕ идёт в OQ

- Локальные задачи → в плане как шаги.
- Системные риски без конкретного вопроса → в RISKS.md.
- Сырые продуктовые сигналы → в IDEAS.md.

## Формат записи

> Поле **Reminder** опционально. Используется для задач которые нельзя делать прямо сейчас, но нужно вернуться к ним при условии (дата, метрика, событие). Без Reminder задача в обычном backlog.
>
> **Expires обязателен если есть Reminder** — без него reminder может висеть Open годами.

```

**Обнаружено через:** /plan для задачи X | /sync-vision | другое
**Тип:** Type C (конфликт vision↔код) | Type E (vision требует уточнения) | pending decision
**Severity:** Blocker | High | Medium | Low
**Статус:** Open | In Discussion | Decided | Implemented | Stale

### Reminder (опционально)

- **Type:** time-gated | metric-gated | event-gated
```

## Жизненный цикл Reminder

| Статус условия | Действие в `/plan` |
|---|---|
| **WAITING** — условие ещё не наступило | Не показывать, пройти мимо |
| **REQUIRES_CHECK** — пора проверить trigger | Показать в Шаге -3.3, обновить `Last checked` |
| **READY** — trigger выполнен, можно работать | Показать как "разблокирован" |
| **EXPIRED** — прошло `Expires`, не закрыто | ⛔ HARD BLOCK до решения PM |

`/retro` Шаг 5а считает здоровье механизма:
- M = количество Reminders ставших READY и обработанных
- N = количество Reminders ставших READY за период
- M/N ≥ 70% — 🟢 здоров; 40-70% — 🟡 нужна корректировка; < 40% — 🔴 не работает.

## Пример (с Reminder)

```
## OQ-005: Удаление legacy-колонок credit_limit/invoicing_type/default_currency из parties

**Дата обнаружения:** 2026-05-14
**Обнаружено через:** /plan Phase D — миграция полей из parties в customers
**Тип:** Type C (конфликт)
**Severity:** Medium
**Статус:** Open

### Reminder

- **Type:** time-gated + metric-gated
- **Trigger:**
  1. Не раньше 2026-06-14 (30 дней compat-period)
  2. AND grep по коду показывает 0 чтений из parties.credit_limit
- **Last checked:** 2026-05-15
- **Expires:** 2026-08-14 (3 месяца — иначе компат-слой остаётся навсегда)

### Источники конфликта

- **Vision/ADR говорит:** ADR-008 — billing fields принадлежат customers, не parties
- **Реальность кода говорит:** erp/Models/Party.php:42 — колонки ещё в schema
- **Inbox говорит:** plan-Phase-D-2026-05-14 — D10 fallback compat, D11 drop через 30 дней

### Варианты решения

1. **Drop колонки + удалить compat-слой** (D11 в плане)
   - Последствия: тесты должны показать что ни один сервис не читает из parties
   - Объём: одна миграция + grep-проверка

2. **Оставить как есть, перевести в legacy archive**
   - Последствия: схема замусоривается, новые разработчики путаются
   - Когда возвращаемся: не возвращаемся

### Рекомендация автора плана

Вариант 1 после 2026-06-14 и подтверждения 0 чтений в логах.

### Связано с

- ADR: ADR-008
- Sync-vision report: 2026-05-14
- DEVLOG: 2026-05-14 [erp-party] Phase D

### Решение PM

[Ожидает]
```
