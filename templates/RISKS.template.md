# RISKS — {{Project Name}}

Pre-mortem: что может пойти не так. Этот документ фиксирует **системные** риски и меры снижения. Обновляется по мере появления новых рисков.

> **Scope (важно):** только системные риски — те что живут месяцами / годами и затрагивают множество задач. Специфичные риски конкретной задачи остаются в плане как мера в фазе, **не** дублировать сюда.

---

## Когда добавлять риск

- Pre-mortem в `/plan` Шаг 98 выявил системную проблему.
- `/sync-vision` обнаружил Type B блокирующий gap.
- `/architecture-audit` показал drift который требует ATTN.
- Inventory кода выявил техдолг с потенциалом большого взрыва.

## Когда НЕ добавлять

- Риск конкретной задачи → план, не RISKS.
- "Учесть потом" без меры → OPEN-QUESTIONS.md или решить сейчас.
- Гипотеза → HYPOTHESES.md.

## Категории

### Риски архитектуры

*Нарушения bounded context, отсутствие изоляции, прямые зависимости между сервисами.*

### Риски документации

*Документация устаревает быстрее кода. Агенты читают устаревшую версию и делают неверные выводы.*

### Риски процесса

*Методологические правила игнорируются, скрипты не запускаются, повторные фиксы одной проблемы.*

### Риски данных и интеграции

*Потеря событий, half-state после миграции, miss-sync с внешними системами, конкурентная запись.*

### Риски безопасности

*Уязвимости, утечки PII, отсутствие audit-log, незащищённые endpoint-ы.*

### Риски производительности

*Деградация под нагрузкой, N+1 запросы, отсутствие индексов, неоптимальные алгоритмы.*

### Риски операционности

*Отсутствие алертов, отсутствие correlation IDs, ручной replay, slow incident response.*

## Активные риски

*Записи добавляются ниже по категориям. Пример:*

### Риски архитектуры

#### R-01 — <Пример>: прямая публикация событий без Outbox (ВЫСОКИЙ)

**Сценарий:** Сервис публикует события напрямую из Model Observers. При сбое broker'а после успешного commit БД — событие теряется. Подписчики не получают уведомления, данные расходятся.
**Текущий статус:** Outbox не реализован (конфликт с ADR-XXX).
**Мера:** Реализовать Outbox в критичных сервисах в первую очередь.

## Архив закрытых рисков

*Закрытые риски перемещаются сюда после ретроспективы. Удалять нельзя — нужны для истории.*

Пример:

### R-05 — <Пример>: Поля domain-логики попадают в общую таблицу (СРЕДНИЙ) ✅ ЗАКРЫТ 2026-05-14

**Сценарий:** Новый разработчик добавляет поле `preferred_carrier` в таблицу `parties` потому что "оно про партнёра". Таблица толстеет, связи теряются.
**Закрыто:** billing fields перенесены из `parties` в `customers` (Phase D, ADR-008). D10 fallback compat → D11 drop через 30 дней.
