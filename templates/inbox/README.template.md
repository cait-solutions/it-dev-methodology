# Inbox — Необработанные артефакты

Сюда кладутся: VCD-документы, ADR-предложения, market signals, анализы конкурентов, экспорт документации из внешних источников, code-derived snapshots — всё что ждёт интеграции в методологию проекта.

---

## Что сюда идёт

- VCD / технические задания от внешних команд
- ADR-предложения от партнёров / контрактов
- Анализы конкурентов
- Дампы документации из чужого репозитория
- Code-derived обзоры (`SERVICE_DOCUMENTATION.md`, `BUSINESS_RULES_MAP.md`)
- Market / domain insights требующие валидации

## Что НЕ сюда

- Срочные баги → issue tracker
- Быстрые вопросы → чат с командой
- Обычные задачи → ROADMAP / task tracker
- Новая фича / идея от пользователя → IDEAS.md

---

## Формат файла

Markdown с любым именем. Желательно — с датой и кратким описанием:

```
2026-05-12-vcd-pricing.md
2026-05-13-competitor-analysis-feature-x.md
2026-05-14-monolith-business-rules-dump.md
```

## Обработка

Inbox-файлы обрабатываются командами `/plan` (Шаг 0.7 — Source Conflict Check) и `/sync-vision`. Любой план задачи затрагивающей данные / контракты / архитектуру **обязан** проверить inbox на релевантные артефакты.

### Шаги обработки

1. Любая команда видящая релевантный inbox-файл → читает его как input.
2. `/plan` Шаг 0.7 классифицирует каждый факт из файла по 5 типам:
   - **Type A** — реальность кода богаче чем описано → обновить vision/ADR
   - **Type B** — описывает несделанное → план / RISKS / OQ
   - **Type C** — конфликт vision↔код → OPEN-QUESTIONS.md
   - **Type D** — источник невалиден / устарел → `_processed/rejected/`
   - **Type E** — vision требует уточнения → OQ + новый ADR
3. После обработки — переместить файл:
   - `_processed/` — обработан, факты интегрированы
   - `_processed/rejected/` — Type D, отвергнут с причиной

### Запись в DEVLOG

При обработке файла — запись в DEVLOG:

```
## YYYY-MM-DD — [inbox-processed] <filename>

**Что:** обработан inbox файл <name>
**Результат:** N Type A → обновлены ADR-X; M Type B → план Phase Y; K Type C → OQ-NNN; ...
**Связано:** [OQ-NNN], [ADR-X], [план YYYY-MM-DD]
```

---

## Стейл-чек

`/sync-vision` Шаг 1 включает проверку: файлы лежащие в inbox > 30 дней без обработки → флаг в отчёте. Скорее всего — не нужны (устарели или дубль), кандидаты на `_processed/rejected/`.

## Структура папки

```
inbox/
├── README.md                    ← этот файл
├── 2026-05-12-vcd-pricing.md    ← необработанный
├── 2026-05-13-analysis.md       ← необработанный
└── _processed/
    ├── 2026-05-01-old-vcd.md    ← обработан, факты в системе
    └── rejected/
        └── 2026-04-15-stale.md  ← Type D, отвергнут (причина в DEVLOG)
```
