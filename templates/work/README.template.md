# work/ — Продукты работы над проектом

Канонический дом для **продуктов работы** (deliverables) — того что **мы произвели**, работая над проектом: research-отчёты (полноразмерные, не строка DEVLOG), аналитика, сгенерированный контент/копирайт, конкурентные разборы, экспортированные датасеты, одноразовые отчёты.

> **Зачем этот дом:** раньше такие артефакты оседали хаотично — `inbox/`, самодельный `docs/content/`, корень репо. Теперь у них один предсказуемый адрес. См. **CLAUDE.md § Artifact Storage Rule**.

---

## Правило в одну строку

**Продукт работы → `work/<stream>/`.** Структура папок = индекс (`ls work/` всегда актуален).

`<stream>` = **направление работы** (например `marketing`, `lead-gen`, `q3-launch`). Один-направленный проект → клади прямо в `work/general/` или плоско в `work/`.

```
work/
├── README.md          ← этот файл
├── <stream>/          ← напр. marketing/ · lead-gen/ · competitor-research/
│   └── <файлы продукта работы>
└── _shared/           ← кросс-stream артефакты (опционально)
```

Подпапки внутри `<stream>/` (`research/`, `deliverables/`, `drafts/`) — **опциональны**, заводи только когда реально нужно разделение. Не плоди пустые папки.

---

## Что сюда идёт

- Полноразмерные research-отчёты (бриф, многостраничный анализ)
- Сгенерированный контент, копирайт, лендинг-тексты
- Конкурентные разборы, аналитика, отчёты
- Экспортированные датасеты, выгрузки

## Что НЕ сюда (граничные различия)

| Артефакт | Где его дом | Почему не `work/` |
|---|---|---|
| Пришло **извне** (VCD, чужой анализ) | `inbox/` | inbox = вход; work = наш выход |
| Durable-**спека** «как устроен продукт» (ADR, design-spec) | `docs/…` | docs = документация системы; work = output работы |
| **Research-вывод** (короткий verdict) | `DEVLOG.md` строка `[research:X]` | у research уже есть дом — не дублировать |
| **Эфемерное** (черновик-превью, `_tmp_*`, промежуточное) | scratchpad / gitignored `_tmp_*` | выбрасываемое, не продукт |
| Living-артефакты методологии (ROADMAP, IDEAS…) | корень / `docs/` | свои канонические дома |

---

## Несколько направлений → когда заводить отдельный workspace

`work/<stream>/` — для **intra-project** продуктов работы. Если направление **«крутится»** как самостоятельный поток (своя дисциплина, свой ритм, свои deliverables регулярно) — рассмотри **promote в отдельный consumer-workspace** (отдельный репо/проект методологии), а не разрастание подпапки. Это согласуется с тем что портфель разнородных направлений = отдельные workspace'ы, не подпапки одного.

---

## Гигиена

- Файлы здесь **трекаются git** (это durable-продукты).
- Эфемерное (`_tmp_*`, `.tmp-*`) в корне репо **не место** — оно gitignored и ловится `validate-work-home.sh` (warn на каждом деплое). Клади его в scratchpad вне репо или удаляй.
