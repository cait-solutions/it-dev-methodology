# HYPOTHESES — methodology-platform

Технические и продуктовые гипотезы + лог их проверок.

Цели:
1. Не принимать непроверенные предположения за факты.
2. Не повторять ошибочные направления при диагностике.

---

## 1. Активные гипотезы (forward-looking)

Гипотезы которые ещё не проверены — нужны эксперименты или данные.

| ID | Гипотеза | Как проверить | Срок | Статус |
|---|---|---|---|---|
| H-001 | <конкретная фальсифицируемая гипотеза> | <метрика / эксперимент> | YYYY-MM-DD | Не начата \| В работе |

### Когда добавлять активную гипотезу

- Принимаешь архитектурное решение на основе предположения ("это должно быть быстрее")
- Выбираешь из нескольких вариантов без реальных данных
- ADR содержит "ОТКРЫТО" — часто это скрытая гипотеза, перенеси сюда

### Что такое хорошая гипотеза

- **Конкретная** — измеримый результат, не "будет работать"
- **Фальсифицируемая** — можно опровергнуть конкретным наблюдением
- **С метрикой** — заранее определено что считаем "подтверждено" vs "опровергнуто"

Плохо: "система будет работать хорошо".
Хорошо: "p95 latency запроса /search останется < 200ms при 10x текущей нагрузке".

## 4. Debug log (backward-looking)

Лог гипотез проверявшихся при диагностике багов. Цель: при следующем похожем баге не повторять опровергнутый путь.

> **Правило:** запись добавляется когда гипотеза опровергнута или подтверждена.
> Не пишем "просто идеи" — только то что проверяли.

### Формат записи

```
## YYYY-MM-DD — [<компонент>]: <краткое описание проблемы>

**Симптом:** что видел пользователь
**Гипотеза:** предполагаемая причина
**Тест:** что сделали чтобы проверить
**Результат:** подтверждено / опровергнуто + данные
**Вывод:** следующий шаг или финальный фикс
```

### Краткий формат для missed-signal

```
## YYYY-MM-DD [missed-signal]
Сигнал: <что>
Почему пропустил: <триггер не сработал / неверная классификация>
Как поймать в следующий раз: <конкретное изменение>
```

---

## Пример debug-записи

```
## 2026-04-28 — [process]: антипетля пропускается под давлением срочности

**Симптом:** За одну сессию 3 деплоя одного компонента за 21 минуту без срабатывания антипетли [fix:N≥2]. Smoke test не выполнен ни после одного.
**Гипотеза:** Под давлением "пользователь ждёт когда заработает" разработчик пропускает процессные шаги. Антипетля как deterrent (не блокер) недостаточно сильна.
**Тест:** 28.04 — 3 sync-фикса за 21 минуту (19:02, 19:14, 19:23), ни один не остановлен антипетлёй.
**Результат:** Подтверждено.
**Вывод:** Антипетля должна стать блокером — требовать явного "это N-й деплой, новые диагностические данные: [X]" перед продолжением. Без признания — деплой блокируется.
```
