# Rules — Tech-stack-specific conventions

Эта папка содержит **методологический гид** по написанию правил технологического стека. Сами правила пишутся в каждом проекте отдельно в `.claude/rules/`.

> Эти файлы (`README.md`, `_TEMPLATE.md`) **не копируются** в проект при bootstrap. Они живут в методологии как референс. Скопируй `_TEMPLATE.md` вручную когда начинаешь писать новый набор правил.

---

## Что такое rules

`.claude/rules/<stack>.md` — это **технологические инварианты** проекта, специфичные для конкретного стека (Laravel, Django, FastAPI, React, PostgreSQL, RabbitMQ, etc.).

Читаются командами `/review` и `/code` при проверке соответствия стандартам.

---

## Чем отличаются от других артефактов

| Артефакт | Уровень | Пример содержания |
|---|---|---|
| `CLAUDE.md` | Операционные правила работы агента | "Реализация только через /code", "Капчер сигналов в IDEAS" |
| `docs/adr/*.md` | Архитектурные решения с обоснованием | "Outbox через таблицу outbox для критичных событий" |
| `docs/vision/*.md` | Стратегия и принципы | "Bounded contexts, изоляция данных по компонентам" |
| `.claude/rules/<stack>.md` | **Стек-конкретные конвенции** | "Primary key — UUID, не autoincrement integer" |

Rules — самый низкий уровень: как именно писать код в этом стеке для этого проекта.

---

## Когда добавлять файл правил

- Проект использует определённый фреймворк / язык / БД.
- Есть конкретные конвенции (отличающиеся от дефолтов фреймворка).
- Есть стек-специфичные анти-паттерны которые проект хочет ловить в `/review`.

## Когда НЕ добавлять

- Правила универсальные → в `CLAUDE.md`.
- Правила архитектурные → в ADR.
- Правила форматирования → в linter config.

---

## Соглашение об именах

`.claude/rules/<stack>.md`, где `<stack>` — нижний регистр, без пробелов:

```
.claude/rules/laravel.md
.claude/rules/postgresql.md
.claude/rules/rabbitmq.md
.claude/rules/react.md
.claude/rules/fastapi.md
.claude/rules/typescript.md
```

---

## Структура файла

Используй [`_TEMPLATE.md`](./_TEMPLATE.md) как стартовую точку. Основные секции:

1. **Модели / схема** — типы PK, обязательные поля, миграции
2. **Сервисный слой** — где живёт бизнес-логика
3. **События / очереди** — pub/sub правила, идемпотентность
4. **Запросы к БД** — N+1, eager loading, raw SQL
5. **Тестирование** — что обязательно тестировать
6. **Конфигурация** — env-переменные, secrets

---

## Как читаются rules командами

- `/code` Шаг 1 — упоминает применимый rules-файл при верификации гипотезы.
- `/review` Шаг 2 — читает `CLAUDE.md` + `.claude/rules/*.md` для контекста проверки.
- Sub-agent `qa` — обязательно читает rules перед проверкой кода.

---

## Жизненный цикл

- Добавление правила → запись в DEVLOG с тегом `[methodology]`.
- Изменение правила — через `/plan` (это методологическое изменение, требует обсуждения).
- Удаление правила — только через `/retro` после анализа что правило не срабатывало 60+ дней.
