# testing-strategy — Стратегия тестирования приложений

Knowledge skill для тестирования приложений которые консьюмер **разрабатывает** (не самой методологии). Цель — обнаружение Frontend и Backend багов: технических (краш, 500, исключение), **логических** (работает, но делает не то), **визуальных** (вёрстка поехала).

> **Принцип (Граница 12):** методология **ведёт** тестирование — даёт знание, помогает агенту сгенерировать и запустить тесты **в консьюмер-проекте**, ведёт регистр найденного. Но она **не исполняет** (нет test-runner движка в платформе — реальный прогон делает агент Claude Code в окружении консьюмера) и **не выносит вердикт о корректности кода** консьюмера (это решение разработчика). Зеркало Оси 6: «ведёт маркетинг, не шлёт письма».

## Tiered model — пирамида под AI-agent workflow

Не каждый тип теста гоняется на каждый коммит. Три уровня по частоте и стоимости:

| Уровень | Когда | Что | Стоимость | Где в методологии |
|---|---|---|---|---|
| **L0 — verify** | каждый `/code` | агент проверяет что изменение работает (1 happy flow, DOM-рендер) | дёшево | `/code` Frontend DOM verification rule (уже существует) |
| **L1 — focused** | `/test` по запросу на затронутую фичу | E2E happy + edge для текущей задачи; новые/недостающие тесты | средне | `/test` команда |
| **L2 — regression suite** | перед prod (pre-release) | весь накопленный suite + visual baseline — «тяжёлая артиллерия» | дорого | `/test regression` (по запросу; блокирующий gate в `/deploy` — отдельная фаза) |

**Принцип:** unit/L0 — быстрые, всегда; E2E/L2 — тяжёлые, редко (перед продом). Это test pyramid (много дешёвых внизу, мало дорогих вверху), адаптированный под то что агент гоняет тесты в консьюмере по запросу, а не CI на каждый push.

## Выбор инструмента per стек

⚠️ **Стек-специфика живёт здесь (в skill) и в `.claude/rules/<stack>.md`, НЕ в команде `/test`** — команда абстрактна (QB8 stack-neutrality, QB10 non-dev parity). Для non-dev / library проектов часть уровней вырождается в N/A.

### Frontend (web app)

| Задача | Инструмент | Зачем |
|---|---|---|
| E2E user-flows | **Playwright** (рекомендуется) / Cypress | реальный браузер, кросс-браузерность, auto-wait. Playwright — быстрее, multi-tab, лучше CI |
| Визуальные регрессии | **Playwright snapshots** (`toHaveScreenshot`) — встроено, бесплатно; **Applitools** (AI visual diff, игнорирует шум) / **Percy** / **Chromatic** (для Storybook) — managed | технический ассерт не ловит «цвет/layout поехал» — нужен pixel/AI diff против baseline |
| Component-level | Vitest + Testing Library / Cypress component | изолированный рендер компонента |

### Backend / API

| Задача | Инструмент | Зачем |
|---|---|---|
| Contract testing | **Pact** (consumer-driven) | ловит расхождение контракта FE↔BE без полной интеграции; контракт генерится из consumer-тестов |
| API fuzzing / schema | **Schemathesis** (property-based из OpenAPI/GraphQL) | авто-генерит тесты из схемы, находит 500/расхождения схемы которые ручные тесты пропускают |
| Property-based (логика) | **Hypothesis** (Python) / fast-check (JS) | вместо примеров — свойства («для любого входа X инвариант Y держится»); ловит edge-входы |
| Integration | нативный фреймворк (pytest / Jest / Go test) + testcontainers | взаимодействие с реальной БД / внешними сервисами |

### Library / CLI / non-dev

- **Library:** unit + property-based; нет E2E/visual (N/A).
- **CLI:** smoke (запуск с флагами → expected output) + snapshot stdout.
- **non-dev workspace** (content, lead-gen): автоматизируемых code-тестов нет → `/test` вырождается в domain-проверку (ссылки валидны? данные кампании консистентны?) или явный N/A.

## Как ловить ЛОГИЧЕСКИЕ баги (не только краши)

Технический ассерт (`expect(res.status).toBe(200)`) ловит краш, но не «кнопка работает, а считает неверно». Подходы:

1. **Property-based testing** — не «вход A → выход B», а «для любого входа инвариант держится» (сумма корзины = Σ позиций; после undo состояние = до действия). Ловит логику на edge-входах которые не придумаешь руками.
2. **Acceptance criteria как тест** — E2E пишется от user-story («пользователь добавил 2 товара по 10 → итого 20, не 200»), проверяет **результат бизнес-логики**, не только что страница отрисовалась.
3. **Schemathesis stateful** — цепочки операций (создать→прочитать→удалить), ловит логику на последовательностях, не одиночных endpoint.
4. **Contract testing** — ловит логическое расхождение «FE ждёт поле `total`, BE шлёт `sum`» — оба работают, но вместе ломаются.

## Визуальные баги — отдельный класс

Технический тест зелёный, но: цвет изменился, элемент наехал, шрифт сломался, кнопка ушла за экран. Только visual diff против **baseline** ловит это:

- **Playwright `toHaveScreenshot`** — pixel diff, baseline в репо. Дёшево, но шумит на анти-алиасинге.
- **AI visual** (Applitools Eyes) — игнорирует шум рендеринга, ловит значимые изменения. Дороже, точнее.
- **Workflow:** первый прогон создаёт baseline → последующие сравнивают → намеренное изменение = обновить baseline (`--update-snapshots`), регрессия = разобраться.

⛔ Без обновления baseline при намеренном UI-изменении — visual тесты дают false positive. Baseline — артефакт, версионируется в консьюмер-репо.

## Регистр найденного — CODE-GAPS.md

Найденные баги (и их статус open/fixed/regression-guard) ведутся в **`CODE-GAPS.md`** консьюмера — отдельный регистр, чтобы разработчик-человек видел баги отдельно, не археологией в DEVLOG.

- **Категории — открытый список** (не закрытый enum — QB10): `frontend-visual`, `frontend-logic`, `backend-contract`, `backend-logic`, `regression`, `perf`, … домен добавляет свои.
- **CODE-GAPS — consumer-owned:** методология даёт шаблон, консьюмер ведёт у себя. Методология **не агрегирует** баги консьюмера в свои артефакты (G-032) — но может читать их read-only для cross-domain pattern detection (`/pull-consumers`).
- **fix-событие → DEVLOG `[fix:X]` + `[test-found:category]`** (тег-указатель) — сохраняет QB3 regression-grep `/review`. CODE-GAPS = регистр со статусом, DEVLOG = событие. Разные роли, ссылаются друг на друга.

**Разграничение артефактов (куда писать):**

| Артефакт | Что фиксирует |
|---|---|
| **CODE-GAPS.md** | product-баг приложения (найден тестом/вручную), его статус open/fixed/regression-guard |
| **AGENT-GAPS.md** | сбой AI-агента (пропустил, ошибся в рассуждении) — не product-баг |
| **RISKS.md** | известная незакрытая **угроза** (ещё не баг, или баг без плана фикса) |
| **HYPOTHESES.md** | гипотеза диагностики с экспериментом |
| **DEVLOG.md** | событие («починили баг X») — `[fix:X]` + `[test-found:category]` |

## Как использовать (направление, не исполнение)

Ты (skill) направляешь агента. Реальный прогон — агент в консьюмер-проекте:

1. **Определи уровень** нужного теста (L0/L1/L2) и стек проекта.
2. **Сгенерируй недостающие тесты** по acceptance criteria задачи (E2E happy + edge; contract если есть API; visual если frontend).
3. **Запусти в консьюмере** (`npx playwright test`, `schemathesis run`, `pytest`) — реальное окружение консьюмера, не методология.
4. **Найденное → CODE-GAPS.md** (статус open) + DEVLOG `[test-found:category]`.
5. **Вердикт о прохождении — разработчику** (Граница 12): сообщи «упало X, продолжить?», не блокируй автоматически (advisory).

При отсутствии фреймворка / браузера / offline → graceful: предложи установку (`npx playwright install`) или explicit skip с письменной причиной (как `/code` DOM rule).

## Что вне scope этого skill

- **Не запускает тесты сам** — направляет агента (исполнение в консьюмере).
- **Не выносит pass/fail вердикт** как блок — это разработчик (Граница 12). Блокирующий L2 gate в `/deploy` — отдельная фаза, разблокируется по подтверждённому сигналу.
- **Не нагрузочное тестирование** (Locust/JMeter/k6) — отдельный домен, упоминается, но не покрыт детально в v1.
