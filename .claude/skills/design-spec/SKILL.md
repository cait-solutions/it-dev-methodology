# design-spec — Создание и обновление Design Spec

**Артефакт:** `docs/services/<service>/<FEATURE>_DESIGN.md` (или `docs/<FEATURE>_DESIGN.md` для cross-service фич)

**Протокол:** основан на VCD v1.51 (Anti-Loss, Draft/Final, колонтитул версий, checklist качества).

**Когда Design Spec, когда ADR:**

| Вопрос | Артефакт |
|---|---|
| Это архитектурное **решение** (принято навсегда, альтернативы отвергнуты)? | ADR |
| Это **спецификация** как именно работает фича/механизм (может уточняться)? | Design Spec |
| Нужна аргументация per-requirement + конкретные примеры? | Design Spec |
| Одно предложение «мы решили X потому что Y, отвергли Z»? | ADR |

**Пример:** ADR-009 (решение: разделить Substitution / BOM / Succession — три механизма, не один) + Design Spec (как именно работает каждый из трёх механизмов с примерами).

## Шаг 1 — Определить режим

Спросить пользователя:

```
Это новый документ или обновление существующего?
  a) Новый Design Spec
  b) Обновление существующего файла
```

**При (b) — сначала прочитать существующий файл целиком** и выполнить Anti-Loss проверку (Шаг 4).

---

## Шаг 2 — Сбор контекста (новый документ)

Задать вопросы итерационно (не все сразу):

1. **Сервис и фича:** Для какого сервиса? Как называется фича/механизм?
2. **Цель и ценность:** Что система должна уметь делать и почему это важно для бизнеса?
3. **Компоненты:** Перечисли 2-5 ключевых подфич / компонентов механизма.
4. **Ограничения:** Есть ли внешние зависимости, стандарты, легаси-ограничения?
5. **Не в scope:** Что явно НЕ входит в эту спецификацию?

Для каждого компонента (из п.3) — собрать:
- Описание (что делает)
- Аргументация (механизм, почему именно так, отвергнутые альтернативы)
- Пример (однострочный сценарий)

**Правило аргументации:** если метрики неизвестны — использовать **High/Medium/Low** или пометку **TBD** с планом измерения. Слова «возможно», «вероятно», «может быть» заменить на **Open Question (OQ-N)**.

## Шаг 3 — Генерация (Draft режим)

Сгенерировать документ по шаблону `templates/DESIGN_SPEC.template.md`:

```markdown
---
title: "<Feature> Design Spec"
service: "<service-name>"
version: "v1.0"
status: "Draft"
created: "<YYYY-MM-DD>"
updated: "<YYYY-MM-DD>"
related_adr: [<список ADR>]
related_plan: "<task-id>"
---

# <Feature> Design Spec — <service-name>

## 1. Цель и ценность
<бизнес-цель одним абзацем>

## 2. Согласованные требования
### 2.N. <Компонент>
- **Описание:** <что>
- **Аргументация:** <механизм, альтернативы, почему>
- **Пример:** <однострочный сценарий>

## 3. Ограничения и допущения
## 4. Не в scope
## 5. Приоритеты реализации
## 6. Открытые вопросы
## 7. Legacy details

---
Previous v. 0.0 = 0 characters
New v. 1.0 = <N> characters
Краткое описание изменений: Initial draft.
```

**Запрет финализации:** автоматически помечать как `status: "Draft"`. Финализация (`status: "Final"`) — только по явной команде пользователя после checklist (Шаг 5).

## Шаг 4 — Anti-Loss (при обновлении)

При обновлении существующего файла — **до любых изменений**:

1. Прочитать текущую версию файла целиком
2. Зафиксировать объём: `Previous v. X.Y = N characters`
3. Определить что планируется изменить / добавить / удалить
4. **Флагировать потери:** если предлагается удалить или сократить подтверждённый контент — явно показать пользователю:
   ```
   ⚠️ Anti-Loss flag: следующий контент будет удалён/сокращён:
      [цитата удаляемого контента]
   Варианты:
     a) Перенести в ## 7. Legacy details
     b) Интегрировать в новую формулировку
     c) Удалить (требует явного подтверждения PO)
   ```
5. Финализация не происходит пока все флаги не разрешены пользователем

**Принцип накопления знаний:** каждая новая версия наращивает и сохраняет полезную информацию предыдущих. Удаление = исключение, не правило.

## Шаг 5 — Checklist качества (перед Final)

Перед переводом в `status: "Final"` — проверить и показать пользователю:

**Качество требований:**
- [ ] Все требования (§2) имеют аргументацию или пометку **TBD** с планом измерения
- [ ] Для каждого требования есть пример (однострочный сценарий)
- [ ] Все конфликты задокументированы как Open Question или решены
- [ ] Раздел §4 «Не в scope» заполнен явным списком
- [ ] Нет слов «возможно», «вероятно», «может быть» без OQ-пометки
- [ ] Связанные ADR указаны в frontmatter (если есть)
- [ ] Колонтитул с diff объёма заполнен

**Верификация реализуемости (Anti-Hallucination gate):**
- [ ] Source каждого механизма §2 явно указан: `real code` (file:line) / `ADR-NNN` / `external doc` (URL/название) / `TBD: OQ-N` — не «по памяти модели»
- [ ] Ни один пункт §2 не противоречит существующим ADR (drift check: `git log -1 --format=%ad <adr-file>` vs код)
- [ ] Каждый пример §2.N реально выполним в текущем стеке (не гипотетический сценарий)
- [ ] §6 Open Questions созданы для всего неизвестного — нет скрытых допущений в тексте §2

Если checklist не пройден — документ остаётся `Draft`.

---

## Шаг 6 — Запись файла

После подтверждения пользователем:

1. Записать файл в `docs/services/<service>/<FEATURE>_DESIGN.md`
2. Добавить запись в DEVLOG: `[feat:design-spec] <service>/<FEATURE>_DESIGN.md v<N> (<Draft|Final>)`
3. Если связан с `/plan` — убедиться что `related_plan` заполнен в frontmatter

**Место хранения:**
- Фича одного сервиса → `docs/services/<service>/<FEATURE>_DESIGN.md`
- Cross-service / платформенная → `docs/architecture/<FEATURE>_DESIGN.md`
- Уже существующий паттерн в проекте → следовать ему (пример: `PRICING_DESIGN.md` в erp-documentantion)
