## Калибровка: что считается осью методологии

1. **Меняет категорию ценности для консьюмера?**
   - "Методология даёт мне дисциплину" → "Методология сама ведёт мой процесс" = ось.
   - "Стало больше команд" = фича.

2. **Через сколько перестанет быть актуальной?**
   - < 6 мес — фича, идёт в ROADMAP.
   - 6-18 мес — среднесрочное направление.
   - 1-3 года — настоящая ось.

3. **Это inherent quality?** (надёжность скриптов, корректность шаблонов) — Quality bar, не ось.

4. **Если убрать ось — продукт остаётся тем же?** Если да — улучшение, не ось.

5. **Открывает новую категорию консьюмеров?** Например мобильные команды / новый технологический стек — да, ось. "ERP-команда станет довольнее" — нет, фича.

## Активные оси

### Ось 1: Self-managing methodology (автозапуск процессов)

*Методология сама ведёт процесс через счётчики и триггеры — не требует от консьюмера помнить когда запустить retro / audit / sync-vision.*

Сейчас система работает в `Phase 1: soft warnings` — `/plan` показывает предложение "запустить /architecture-audit?" но консьюмер может пропустить. После 1-2 месяцев накопления данных в `/retro` мы решим какие триггеры перевести в `Phase 2: hard blocks`. Цель — минимизировать когнитивную нагрузку владельца проекта: методология сама знает что пора, и блокирует продвижение пока не сделано.

**Что открывает:** консьюмеры не дрейфуют. Архитектурная документация всегда актуальна потому что без `/architecture-audit` нельзя стартовать новый план. Risks не накапливаются потому что `/retro` обязателен каждые 15 планов.

**Временной горизонт:** 2 года. Ось перестанет быть актуальной если Claude Code сам начнёт детектить когда методологическая команда нужна (например по diff кода или по логам). Сейчас детект на стороне `/plan` Шаг -3 — но это level-2 регулятор.

**Сигналы:** `/retro` skip-rates показывают какие триггеры консьюмеры игнорируют. ERP уже использует triggers.json по полной — даст первые данные через 2-3 месяца.

**Стоимость:** изменения в каждой команде; новая логика в `/plan` для hard-block перехода; миграция consumers на новую схему.

**DEVLOG тег:** `[feat:self-managing]`

### Ось 2: Stack-agnostic adoption *(новая в v2)*

*Методология работает на любом технологическом стеке без переписывания core. Один и тот же `/plan`, `/code`, `/review` процесс для PHP/Laravel команды и Python/FastAPI проекта.*

Stack-specific знания живут в `.claude/rules/<stack>.md` на стороне консьюмера, не в методологии. Это позволяет добавлять новые проекты cait.solutions с любым стеком без unfork-fork методологии и без отдельной версии для каждой команды.

Сейчас инфраструктура для этого есть — папка `rules/` с гидом и `_TEMPLATE.md`. Но это **не озвучено как ось** — выглядит как побочная фича. Сделать осью значит: (1) создать каталог "starter rules" (`rules/starters/laravel.md`, `rules/starters/django.md`, `rules/starters/fastapi.md`, etc.) с готовыми наборами под популярные стеки cait.solutions, (2) добавить в `new-project-init.sh` флаг `--starter-rules=<stack>` копирующий релевантный starter в `.claude/rules/`, (3) задокументировать в `CLAUDE.template.md` как `rules/` работает.

**Что открывает:** новые консьюмеры внутри cait.solutions без барьера входа. Команда добавляющая проект на новом стеке (например Go или Kotlin) использует ту же методологию без переписывания.

**Временной горизонт:** 1-2 года. Перестанет быть актуальной если все проекты cait.solutions перейдут на один стек (маловероятно).

**Сигналы:** PAI на Python (Telegram bot + Gemini SDK), ERP на PHP/Laravel. Уже два разных стека работают через единые команды — реально, но без формализации starter rules.

**Стоимость:** дисциплина — никакой stack-specific логики в `commands/`, `templates/`, `hooks/`. Maintenance starter rules для популярных стеков (без поддержки они устаревают).

**DEVLOG тег:** `[feat:stack-agnostic]`

### Ось 3: Cross-project standardization

*Один источник правды на slash-команды и шаблоны артефактов. Никакого PAI-стиля vs ERP-стиля — единая методология которая адаптируется под размер проекта через флаги.*

После Phase A-F эта ось **в основном реализована**. PAI и ERP станут консьюмерами одной методологии. Дальше — поддерживать инвариант: новые паттерны идут через PR в этот репо, а не локальной правкой консьюмера.

Следующий уровень: автоматизированный version drift check. Когда консьюмер прогоняет `/plan`, проверяется `.claude/.version` против HEAD методологии (через `git fetch`); если расхождение > N версий — WARNING.

**Что открывает:** новый разработчик может перейти между проектами cait.solutions не переучивая методологию. Знания о `/plan`, формате DEVLOG, ROADMAP-секциях — переносимые.

**Временной горизонт:** 1-2 года. Перестанет быть актуальной если каждый проект эволюционирует так далеко в своих нуждах, что один шаблон не покрывает. Тогда — разветвление методологии по типам (`platform-methodology` vs `library-methodology` vs ...).

**Сигналы:** запросы консьюмеров на новые опциональные шаблоны (текущие `--with-X` флаги отвечают на этот сигнал).

**Стоимость:** дисциплина владельца методологии — никогда не принимать локальные правки без upstream PR.

**DEVLOG тег:** `[feat:standardization]`

---

### Ось 4: Dog-fooding (методология сама использует свою методологию)

*Изменения в этой методологии идут через её собственный `/plan` → `/code` → `/review` → `/deploy`. Phase F запустил эту ось; v2 этого VISION — первое реальное применение через `/product-vision`.*

С этого момента любая новая команда / шаблон / правило проходит через тот же процесс который консьюмеры применяют к своим проектам. Если процесс плохо работает на самом методологическом репо — значит он плохо работает и у консьюмеров. Прямая обратная связь.

⚠️ *Гипотеза, требует валидации:* "Self-application уменьшит дрейф между задуманной и реальной работой методологии." Основано на интуиции "eat your own dog food", не подтверждено эмпирически. Через 2-3 месяца увидим что больше срабатывает: `/plan` процесс или прямые правки.

**Что открывает:** доверие к методологии. Если она хорошо работает на меня самого — значит работает.

**Временной горизонт:** постоянно. Эта ось не имеет obsolescence условия — это операционная практика.

**Сигналы:** какие команды чаще пропускаются на этом репо? Что заставляет редактировать напрямую без `/plan`? Этот первый `/product-vision` уже даёт сигнал — насколько громоздкий процесс ощущается.

**Стоимость:** нулевая инфраструктурная (всё есть после Phase F). Требует только дисциплины владельца.

**DEVLOG тег:** `[feat:dog-fooding]`

## Watch list (отложенные оси)

*Кандидаты на оси которые прошли 5-вопросный фильтр, но сейчас не активируются — нужны конкретные данные или сигналы.*

### Engineering analytics

DEVLOG, `triggers.json`, IDEAS.md консьюмеров — это датасет о поведении методологии. Через 6 месяцев накопления станет возможно: какие команды чаще пропускаются, какие триггеры шумят (skip-rate > 50%), какой % планов доходит до `/code`, какие классы багов повторяются между проектами.

**Триггер активации:** 3+ месяца `triggers.json` data + 30+ DEVLOG entries у консьюмеров.
**Strategic importance:** HIGH через год-два.
**Текущий leverage:** LOW — данных нет.

### Multi-tool / multi-LLM support

Slash-команды → universal prompt templates работающие в любом AI-tool (Cursor, Aider, Copilot Workspace, Cody). Сейчас методология Claude-Code-specific.

**Триггер активации:** сигнал что Claude Code теряет позиции ИЛИ у cait.solutions появится команда требующая другой инструмент.
**Strategic importance:** HIGH (survival hedge — если Anthropic ослабнет, методология сейчас умирает с Claude Code).
**Текущий leverage:** LOW — нет сигнала, огромная стоимость переписывания.

### Cross-project knowledge propagation

Уроки в DEVLOG одного проекта автоматически попадают в IDEAS.md другого когда они релевантны (через keyword/semantic matching).

**Триггер активации:** 50+ DEVLOG entries в нескольких проектах одновременно + появятся очевидные cross-project паттерны.
**Strategic importance:** MEDIUM (улучшает методологию как learning organism).
**Текущий leverage:** LOW — уроки сейчас разные.

## Quality bar (inherent qualities — не оси)

Операционные требования. Не направления развития, но без них методология не работает.

### 1. Корректность скриптов

`new-project-init.sh` и `sync-methodology.sh` идемпотентны (повторный запуск = тот же результат) и не ломают существующее state консьюмера. Любой push в main должен сопровождаться тестом обоих скриптов end-to-end.

*Work items:* CI который запускает bootstrap → sync → проверяет diff = пусто.
*Статус:* в ROADMAP → Considered.

### 2. Корректность шаблонов

`{{Project Name}}` плейсхолдер не должен оставаться после bootstrap. Все cross-ссылки между артефактами должны резолвиться. VERSION bump при каждом изменении синхронизируемого артефакта.

*Work items:* lint-скрипт проходящий по всем шаблонам и проверяющий (а) валидность markdown, (б) отсутствие неподставленных placeholders в выходе bootstrap, (в) рабочие ссылки.

### 3. Regression prevention *(новый в v2)*

Команды (особенно `/review` и `/code`) обязательно содержат class-bug detection: grep на параллельные пути в проекте, проверка что аналогичный фикс применён везде.

*Симптом нарушения:* один тип бага исправлен в одном файле, `/review` не пойман в параллельном файле. Class-bug-rule в `commands/review.md` Шаг 3 — это операционализация этого Quality bar.

### 4. Security awareness *(новый в v2)*

Все `[security]` задачи проходят через `security` sub-agent. Любой новый endpoint / интеграция с PII / изменение auth логики требует обновлённый `docs/threat-model-*.md`.

*Симптом нарушения:* появился новый endpoint без записи в threat-model. `/review` Шаг 3 (Безопасность checklist) обязателен для всех `[security]` и `[infra]` задач.

### 5. Edge case detection *(новый в v2)*

`/plan` Шаг 98 (Pre-Mortem с 6 сценариями) обязателен для Full mode. Не пропускается даже под давлением срочности.

*Симптом нарушения:* pre-mortem пропускается, после деплоя обнаруживается случай не из плана. **Принудительный триггер уже в команде:** если пользователь выразил сомнение ("точно проверил?") — pre-mortem обязателен в любом режиме.

### 6. Closed product feedback loop *(новый в v2)*

Цикл `IDEAS → /product-review → ROADMAP → /plan → DEVLOG → /retro → IDEAS` не должен прерываться. **Ни один сигнал не теряется**: либо в IDEAS, либо явно отвергнут с причиной.

*Симптом нарушения:* разработчик упомянул "пользователь жаловался на X" в коммите, но в IDEAS этого сигнала нет — методология упустила. Capture-правило в CLAUDE.md (silent IDEAS.md adds) — операционализация этого Quality bar.

## Стратегические границы

*Что методология не есть и не станет — снимается только явным решением, не дрейфом.*

### 1. Только владелец контролирует канон *(новый в v2)*

Read-only для разработчиков консьюмер-проектов. Локальные правки запрещены — все изменения через PR в `methodology-platform`. Снимается только если cait.solutions решит открыть методологию как community-driven проект (open-source).

Сейчас фактически работает через banner-механизм + warning при sync. v2 формализует это как стратегический выбор, а не побочный эффект архитектуры.

### 2. Не плагин-система — монолит с флагами *(новый в v2)*

Опциональные шаблоны через `--with-X` флаги в `new-project-init.sh`, не через рантайм-плагины загружаемые из кода консьюмера. Сложность плагин-системы (versioning, isolation, API stability) не оправдана для проекта внутри одной компании.

### 3. Не runtime-система

Нет процессов которые методология запускает в production консьюмера. Только скрипты для bootstrap/sync + slash-команды для Claude Code. `/deploy` — это чек-лист и DEVLOG-формат, не deployment engine.

### 4. Не валидирует код консьюмера

Это работа линтеров и type checker'ов. Методология валидирует **процесс** — запустили ли `/review`, есть ли DEVLOG-запись, обновлён ли SYSTEM-MAP.

### 5. Не автоматизирует sync

Каждый консьюмер вручную решает когда подтягивать обновления через `sync-methodology.sh`. Авто-pull (cron, CI) — out of scope: нарушает контроль владельца проекта над тем когда обновляться.

### 6. Не разветвляется по типу проекта

Один шаблон, флаги для опциональных компонентов. `rules/` живёт в каждом проекте, не в методологии (после имплементации Оси 2 — starter rules в методологии, но как стартовая точка, не как ветвление).

Если появится класс проектов которые принципиально не вписываются (например mobile с принципиально другим build flow) — рассматривается split методологии, но это требует явного решения.
