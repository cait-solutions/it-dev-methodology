# how — Единый вход-роутер по задачам

Точка входа когда пользователь не знает **с какой команды начать** или **как подойти к задаче**. Вызывается явно (`/how` / `/how <описание задачи>`) ИЛИ авто-активируется на естественном языке («с чего начать», «как мне X», «how do I»). Не заменяет команды — помогает выбрать правильную точку входа.

**Принцип (анти-drift by design):** routes by **task type**, not by command list. Роутинг идёт по стабильной онтологии из ~6 типов работы (research / plan / implement / audit / diagnose / strategy) — она **не растёт** с числом команд. Новая команда попадает в существующий тип → этот skill её уже покрывает **без правки**. Именно поэтому здесь НЕТ перечня всех команд и НЕТ чтения их списка — список устаревает, тип работы нет. Это и есть механизм, гарантирующий что роутер не протухает.

---

## НЕ активировать при:

- Пользователь уже написал `/plan`, `/code`, `/diagnose` или другую команду — она уже выбрана.
- Вопрос типа «как сделать X в коде» с конкретным техническим ответом.
- Уже идёт работа в рамках запущенной команды (mid-task вопрос).
- Баг с понятным симптомом → `/diagnose` или `/code` напрямую.
- Маркетинговая / SEO / визуальная задача → свои skills (маркетинг, seo-audit, web-design).

---

## Шаг 0 — Inventory: учитывай ВСЕ доступные в проекте команды и skills (не только методологические)

Таблица routing в Шаге 2 покрывает **методологические** команды (стабильны, синкаются всем). Но конкретный проект-консьюмер может иметь **собственные локальные** skills/команды, которых методология не знает (напр. `verify-ai-visual`, `verify-ai-endpoint` — project-scoped ops-skills в ERP, `scope: project`, без methodology-banner). Роутер обязан их учитывать — иначе направит на общий `/test`, когда у проекта есть более точный локальный инструмент.

**Механизм (zero-cost — не читать файлы циклом):** описания всех доступных в этой сессии skills уже видны тебе в контексте (именно так работает их auto-активация). Поэтому:
1. При выборе рекомендации **просканируй доступные skills/команды глазами** — включая те, что НЕ в таблице Шага 2 (признак локального: `scope: project` / нет строки banner «Synced from methodology-platform» / имя, которого нет в методологии).
2. Если локальный skill/команда **точнее** покрывает задачу пользователя, чем методологический дефолт — **рекомендуй его**, назвав явно как project-local («у твоего проекта есть `<name>` — он подходит лучше `/test` для этого»).
3. Если сомневаешься что доступно — быстрый `ls .claude/skills/ .claude/commands/` в проекте-консьюмере покажет фактический набор (методологические + локальные вперемешку).

⛔ Не ограничивай рекомендацию таблицей Шага 2 — она про методологию; проект может уметь больше. Task-type routing (Шаг 2) остаётся быстрым путём для методологических команд, локальные капабилити — поверх него.

## Шаг 1 — Определить тип работы

Задай один вопрос если неясно: **«Это исследование, планирование, реализация, аудит или диагностика проблемы?»**

Если ответ понятен из контекста — перейди сразу к Шагу 2.

---

## Шаг 2 — Routing по типу задачи

### 🔍 Исследование / «Не знаю как подойти к X»

> Нужно понять предметную область, изучить существующий код, собрать требования перед планированием.

**Рекомендация:** начни с `/research` (внешние факты) или Agent/Explore (анализ существующего кода).  
**Потом:** → `/plan`  
**Модель:** Default (Sonnet) · effort: High · thinking: ON

**Сигнал:** «изучи Party-компонент», «что используют конкуренты», «как устроен этот модуль»

---

### 📋 Планирование / «Хочу сделать X, как лучше подойти»

> Есть цель, нужно определить архитектуру, риски, порядок шагов.

**Рекомендация:** `/plan`  
**Предварительно:** если цель касается UI/UX — прочитай `/web-design` (aesthetic direction ДО планирования); если нужна спецификация механизма — `/design-spec`  
**Если решение high-stakes** (новый механизм, breaking change, затрагивает ≥3 компонентов): запусти `/opinion` ПЕРЕД `/code` — council 7/7 проверит правильность direction до реализации  
**Модель:** Default (Sonnet) · effort: High · thinking: ON

**Сигнал:** «хочу добавить модуль», «как организовать интеграцию», «план рефакторинга»

---

### ⚙️ Реализация / «Знаю что делать, нужно сделать»

> Есть план или конкретная задача, нужно писать код.

**Рекомендация:** `/code`  
**Предварительно:** убедись что `/plan` уже был (или задача simple/Lite). Если нет — вернись к Планированию.  
**Модель:** Default (Sonnet) · effort: Medium · thinking: OFF (Lite); High · thinking: ON (Full, архитектурные изменения)

**Сигнал:** «реализуй план», «напиши компонент X», «добавь поле в форму»

---

### 🔬 Аудит / «Что не так с архитектурой / документацией / тестами»

> Нужно проверить текущее состояние без конкретного fix.

| Что проверяешь | Команда |
|---|---|
| Архитектура и компоненты | `/architecture-audit` |
| Документация актуальна? | `/doc-audit` |
| Методология у консьюмеров актуальна? | `/push-consumers` (maintainer доставляет, push-only) · healthcheck: `scripts/sync-doctor.sh` |
| Тестирование приложения (баги: технические/логические/визуальные) | `/test` (L1 focused · `/test regression` · `/test visual`) |
| Качество UX существующих флоу (friction vs эталон) | `/ux-audit` |
| PR перед мержем | `/review` |

**Frontend-ось (баги vs качество — разные вещи, разный output):** «сломалось ли / вёрстка поехала / логика неверна» → `/test` (пишет баги в CODE-GAPS). «Хорош ли сам флоу, удобно ли, на бренде ли» → `/ux-audit` (приоритизированные находки → `/plan`). Идут последовательно: сначала `/test` (чинить баги), потом `/ux-audit` (оценивать удобство на рабочем флоу).

**Модель:** Default (Sonnet) · effort: High · thinking: ON

---

### 🐛 Диагностика / «Что-то сломалось / не работает»

> Есть симптом, нужно найти причину.

**Рекомендация:** `/diagnose`  
**Если уже знаешь причину:** → `/code` напрямую  
**Если повторяющаяся проблема (3-й фикс одного домена):** `/diagnose` обязателен перед `/code`  
**Модель:** Default (Sonnet) · effort: High · thinking: ON

**Сигнал:** «не работает», «падает», «регрессия после деплоя»

---

### 🎯 Стратегия / «Что строить дальше»

> Нужно расставить приоритеты или переосмыслить direction.

| Что нужно | Команда |
|---|---|
| Приоритизация задач по ценности | `/roadmap` |
| Честное мнение по решению/подходу | `/opinion` |
| Ретроспектива | `/retro` |
| Продуктовая стратегия | `/vision` |

**Модель:** Capable (Opus) · effort: High · thinking: ON (для `/opinion`, `/vision`)

---

## Шаг 3 — Показать рекомендацию

Вывести:
1. Тип задачи (из Шага 1)
2. Рекомендованная команда с обоснованием (1-2 предложения)
3. Рекомендованная модель в трёхмерном формате (`tier · effort · thinking`)
4. Если нужен предварительный шаг — назвать его явно

**Не перечислять все команды** — только релевантные для текущей задачи.

## Примеры routing

| Задача пользователя | Тип | Рекомендация |
|---|---|---|
| «Хочу переделать фронтенд ERP, с чего начать?» | Исследование → Планирование | Сначала Agent/Explore (анализ существующего кода) → `/plan` |
| «Как организовать интеграцию с CRM?» | Планирование | `/plan` · High · ON; если high-stakes → `/opinion` перед `/code` |
| «Реализуй то что мы спланировали» | Реализация | `/code` · Medium · OFF |
| «Что-то сломалось после последнего деплоя» | Диагностика | `/diagnose` |
| «Стоит ли разделять этот модуль на два?» | Стратегия / Opinion | `/opinion` · Capable · High · ON |
| «Архитектура устарела, нужно проверить» | Аудит | `/architecture-audit` |
