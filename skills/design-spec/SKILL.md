# design-spec — Создание и обновление Design Spec

**Артефакт:** `docs/services/<service>/<FEATURE>_DESIGN.md` (или `docs/<FEATURE>_DESIGN.md` для cross-service фич)

**Протокол:** основан на VCD v1.51 (Anti-Loss, Draft/Final, колонтитул версий, checklist качества).

**Когда Design Spec, когда ADR:**

| Вопрос | Артефакт |
|---|---|
| Это архитектурное **решение** (принято навсегда, альтернативы отвергнуты)? | ADR |
| Это **спецификация** как именно работает фича/механизм (может уточняться)? | Design Spec |
| Нужна аргументация per-requirement + конкретные примеры? | Design Spec |
| Одно предложение «мы решили X потому что Y, отвергли Z»? | ADR |

**Пример:** ADR-009 (решение: разделить Substitution / BOM / Succession — три механизма, не один) + Design Spec (как именно работает каждый из трёх механизмов с примерами).

## Шаг 1 — Определить режим

Спросить пользователя:

```
Это новый документ или обновление существующего?
  a) Новый Design Spec
  b) Обновление существующего файла
```

**При (b) — сначала прочитать существующий файл целиком** и выполнить Anti-Loss проверку (Шаг 4).

**Context-entry mode detection (SYS-002, closes G-104):** ПЕРЕД переходом к Шагу 2 — оценить контекст текущего запроса:

- **Interview mode** (стандарт): пользователь ещё не предоставил контекст → задавать вопросы Шага 2 итерационно.
- **Synth mode** (closes G-104): пользователь уже предоставил **всю ключевую информацию** (сервис + фича + компоненты + ограничения) в запросе — либо явно через готовый текст, либо через PRODUCT.md / design-doc / PRD. Признак: ≥ 3 пункта из Шага 2 уже известны из запроса.

При **synth mode** — предложить пользователю:
```
Я вижу что контекст уже есть: <сервис>, <фича>, <N компонентов>.
Перейти сразу к генерации (synth mode) или уточнить дополнительно? (synth / уточнить)
```
При `synth` — пропустить Шаг 2 и перейти к Шагу 3 напрямую с имеющимся контекстом.

## Шаг 2 — Сбор контекста (новый документ)

Задать вопросы итерационно (не все сразу):

1. **Сервис и фича:** Для какого сервиса? Как называется фича/механизм?
2. **Цель и ценность:** Что система должна уметь делать и почему это важно для бизнеса?
3. **Компоненты:** Перечисли 2-5 ключевых подфич / компонентов механизма.
4. **Ограничения:** Есть ли внешние зависимости, стандарты, легаси-ограничения?
5. **Не в scope:** Что явно НЕ входит в эту спецификацию?

Для каждого компонента (из п.3) — собрать:
- Описание (что делает)
- Аргументация (механизм, почему именно так, отвергнутые альтернативы)
- Пример (однострочный сценарий)

**Правило аргументации:** если метрики неизвестны — использовать **High/Medium/Low** или пометку **TBD** с планом измерения. Слова «возможно», «вероятно», «может быть» заменить на **Open Question (OQ-N)**.

### Scope-gate (Closes G-120 / P-012)

После получения списка компонентов — **до начала генерации** — выполнить проверку scope:

**Критерии рекомендации декомпозиции:**

1. **Cross-domain граница (primary):** компоненты §2 принадлежат разным доменам с независимыми жизненными циклами (например: ценообразование + генератор номеров + типы статей — три разных бизнес-домена). Каждый домен = отдельный Design Spec.
2. **≥ 6 независимых компонентов §2 (secondary signal):** сигнал что граница может быть размыта. Для 5 тесно связанных компонентов одного домена (например: Substitution / BOM / Succession — все про замены одной сущности) — декомпозиция не нужна.

**Подтверждающий симптом (не критерий):** прогнозируемый объём > 400 строк. Это следствие, не причина.

**Алгоритм:**

```
Есть cross-domain граница?
  ДА → рекомендация декомпозиции (обязательно)
  НЕТ → ≥ 6 компонентов?
           ДА → рекомендация декомпозиции (мягко)
           НЕТ → продолжить без декомпозиции
```

**Формат рекомендации (recommendation-first):**

```
📋 Scope check: выявлено <N> компонентов / cross-domain граница <описание>.

Рекомендую разбить на отдельные Design Spec:
  1. <Spec A> — компоненты: <2.1, 2.2, 2.3>
  2. <Spec B> — компоненты: <2.4, 2.5>

Обоснование: <одна строка — почему граница именно здесь>.

Варианты:
  a) Принять декомпозицию → начнём с <Spec A>
  b) Оставить одним документом → продолжаем
```

Агент **не блокирует** создание единого документа если пользователь выбрал (b). Рекомендация — не запрет.

## Шаг 3 — Генерация (Draft режим)

Сгенерировать документ по шаблону `templates/DESIGN_SPEC.template.md`:

```markdown
---
title: "<Feature> Design Spec"
service: "<service-name>"
version: "v1.0"
status: "Draft"
created: "<YYYY-MM-DD>"
updated: "<YYYY-MM-DD>"
related_adr: [<список ADR>]
related_plan: "<task-id>"
---

# <Feature> Design Spec — <service-name>

## 1. Цель и ценность
<бизнес-цель одним абзацем>

## 2. Согласованные требования
### 2.N. <Компонент>
- **Описание:** <что>
- **Аргументация:** <механизм, альтернативы, почему>
- **Пример:** <однострочный сценарий>
- **Source:** [real code file:line | ADR-NNN | external doc | TBD: verify before Final (OQ-N)]
- **Диаграмма:** <mermaid или skip-причина — см. правило ниже>

## 3. Ограничения и допущения
## 4. Не в scope
## 5. Приоритеты реализации
## 6. Открытые вопросы
## 7. Legacy details

---
Previous v. 0.0 = 0 characters
New v. 1.0 = <N> characters
Краткое описание изменений: Initial draft.
```

**Запрет финализации:** автоматически помечать как `status: "Draft"`. Финализация (`status: "Final"`) — только по явной команде пользователя после checklist (Шаг 5).

**Early source + ADR check (SYS-002, closes pre-Шаг-5 drift):** при генерации каждого §2.N — заполнить поле `Source:` немедленно, не откладывать на Шаг 5. Если механизм §2.N очевидно противоречит известному ADR → создать Open Question сразу (`OQ-N: конфликт с ADR-NNN — уточнить перед Final`), не ждать финализации. Шаг 5 Anti-Hallucination gate **подтверждает** источники, а не обнаруживает их впервые.

### Правило per-component диаграмм (Closes G-120)

**Для каждого `§2.N` компонента** — добавить диаграмму если компонент **нетривиальный**:

**Критерий нетривиальности** (достаточно одного):
- Поток с ветвлением (пользователь делает X → система выбирает между A и B)
- Состояния (объект переходит между состояниями: Draft → Review → Published)
- ≥ 3 последовательных шага взаимодействия (User → SvcA → SvcB → DB)

**Если нетривиальный:** сгенерировать mermaid-блок прямо в `§2.N`. Тип диаграммы:
- `flowchart TD` — для пользовательского потока / ветвления решений
- `stateDiagram-v2` — для машин состояний
- `sequenceDiagram` — для multi-service взаимодействия

**Node format (CLAUDE.md §3 «Формат node-описания», v5.57.0):** компонентные ноды в flowchart-диаграммах §2.N и §8 — три строки понятные нетехническому читателю:
`NodeID["🔹 Имя<br/>Зачем: назначение<br/>Без него: что сломается"]`. Affordance-ноды освобождены.

Формат слота в `§2.N`:

```markdown

После записи файла — выполнить: `bash scripts/update-mermaid-links.sh <path-to-file>` (заменит placeholder на реальный URL).

**⛔ G-100 запрет:** pako-URL НЕ генерировать в чате. Ссылку пишет скрипт в файл — агент даёт только `[file:line](path#Lline)` в ответ пользователю.

**Если тривиальный** (простое поле, флаг, lookup без ветвления): вместо диаграммы написать:

```markdown
**Диаграмма:** _нет — компонент тривиальный (нет ветвления/состояний)_
```

### §8 Сквозной сценарий — когда обязателен / когда skip

**Обязателен (§8 = Обязательная):** если ≥ 2 компонентов §2 взаимодействуют в одном потоке (выход одного — вход другого) И это взаимодействие не видно из отдельных диаграмм §2.N.

**Рекомендован (§8 = Рекомендуется):** для презентации PO/stakeholder, онбординга (3+ компонента, независимые но связанные в happy path).

**Skip (§8 не нужен):** ≤ 2 компонента без сквозного взаимодействия; каждый §2.N диаграммой уже показывает полный поток своего компонента.

Указать в документе явно какой вариант и почему:
```markdown
## 8. Сквозной сценарий
> **Статус §8:** Обязательна / Рекомендуется / Skip — <одна строка причина>
```

## Шаг 4 — Anti-Loss (при обновлении)

При обновлении существующего файла — **до любых изменений**:

1. Прочитать текущую версию файла целиком
2. Зафиксировать объём: `Previous v. X.Y = N characters`
3. Определить что планируется изменить / добавить / удалить
4. **Флагировать потери:** если предлагается удалить или сократить подтверждённый контент — явно показать пользователю:
   ```
   ⚠️ Anti-Loss flag: следующий контент будет удалён/сокращён:
      [цитата удаляемого контента]
   Варианты:
     a) Перенести в ## 7. Legacy details
     b) Интегрировать в новую формулировку
     c) Удалить (требует явного подтверждения PO)
   ```
5. Финализация не происходит пока все флаги не разрешены пользователем

**Принцип накопления знаний:** каждая новая версия наращивает и сохраняет полезную информацию предыдущих. Удаление = исключение, не правило.

## Шаг 5 — Checklist качества (перед Final)

Перед переводом в `status: "Final"` — проверить и показать пользователю:

**Качество требований:**
- [ ] Все требования (§2) имеют аргументацию или пометку **TBD** с планом измерения
- [ ] Для каждого требования есть пример (однострочный сценарий)
- [ ] Все конфликты задокументированы как Open Question или решены
- [ ] Раздел §4 «Не в scope» заполнен явным списком
- [ ] Нет слов «возможно», «вероятно», «может быть» без OQ-пометки
- [ ] Связанные ADR указаны в frontmatter (если есть)
- [ ] Колонтитул с diff объёма заполнен

**Верификация реализуемости (Anti-Hallucination gate):**
- [ ] Source каждого механизма §2 явно указан: `real code` (file:line) / `ADR-NNN` / `external doc` (URL/название) / `TBD: OQ-N` — не «по памяти модели»
- [ ] Ни один пункт §2 не противоречит существующим ADR (drift check: `git log -1 --format=%ad <adr-file>` vs код)
- [ ] Каждый пример §2.N реально выполним в текущем стеке (не гипотетический сценарий)
- [ ] §6 Open Questions созданы для всего неизвестного — нет скрытых допущений в тексте §2

**Диаграммы (Closes G-120 + G-121):**
- [ ] Каждый нетривиальный §2.N имеет mermaid-диаграмму **или** явную skip-причину (`_нет — компонент тривиальный_`)
- [ ] Ни один mermaid-блок не содержит `PLACEHOLDER` (незаполненный placeholder = диаграммы нет)
- [ ] После записи файла выполнен `bash scripts/update-mermaid-links.sh <path>` — все ссылки актуальны
- [ ] §8 статус явно указан: `Обязательна` / `Рекомендуется` / `Skip` с одной строкой причины
- [ ] Компонентные ноды в flowchart-диаграммах §2.N и §8 следуют формату **Имя + Зачем + Без него** (≥ 2 `<br/>` на ноду) согласно CLAUDE.md §3 «Формат node-описания» (v5.57.0)

**⛔ Блок Final:** если хотя бы один нетривиальный §2.N не имеет диаграммы И нет skip-причины — документ остаётся `Draft`.

Если checklist не пройден — документ остаётся `Draft`.

## Шаг 6 — Запись файла

После подтверждения пользователем:

1. Записать файл в `docs/services/<service>/<FEATURE>_DESIGN.md`
2. Добавить запись в DEVLOG: `[feat:design-spec] <service>/<FEATURE>_DESIGN.md v<N> (<Draft|Final>)`
3. Если связан с `/plan` — убедиться что `related_plan` заполнен в frontmatter

**Место хранения:**
- Фича одного сервиса → `docs/services/<service>/<FEATURE>_DESIGN.md`
- Cross-service / платформенная → `docs/architecture/<FEATURE>_DESIGN.md`
- Уже существующий паттерн в проекте → следовать ему (пример: `PRICING_DESIGN.md` в erp-documentantion)
